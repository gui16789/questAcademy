## 2. 文档目的

本文档定义《动物侦探城》的项目级架构，重点回答：

- 后续是否支持切换教材版本。
- 知识点如何从教材中抽取、维护和复用。
- 内容工厂如何生产题目、知识卡、线索、关卡和 Boss。
- 游戏化学习流程如何由知识体系编排出来。
- 学习数据如何反向支持错题诊断、掌握状态和复习推荐。
- MVP 如何作为整体架构下的第一条最小实现路径，而不是一次性写死的原型。

本文档位于 PRD 之前。PRD 定义某个版本做什么；本文档定义整个系统应该如何长期生长。

## 3. 架构总原则

### 3.1 核心判断

```text
知识体系是内核，案件是外壳；教材和知识点驱动内容，游戏流程负责包装和激励。
```

系统不能把“徽章失踪案”或“除法题目”写成唯一主线。正确的关系应该是：

```text
教材知识体系
-> 知识点地图
-> 内容资产
-> 案件与关卡编排
-> 学习体验
-> 学习数据
-> 复习与诊断
```

也就是说：

```text
不是知识点从案件里长出来，
而是案件从知识点地图里组装出来。
```

### 3.2 设计原则

| 原则 | 说明 |
| --- | --- |
| 教材可切换 | 教材版本、年级、学期、单元不能写死在代码和游戏流程中 |
| 知识点可维护 | 知识点、能力点、易错点、前置知识需要结构化 |
| 内容可生产 | 题目、解析、知识卡、线索、Boss 应能按模板生产和审核 |
| 游戏可编排 | 案件、关卡、Boss 是对知识路径的游戏化编排 |
| 数据可诊断 | 答题记录必须能回到知识点、能力点和错因 |
| 后台可演进 | MVP 可静态配置，后续应升级为内容管理后台 |
| 审核可追踪 | 内容上线需要教研审核、儿童可读性审核和版本记录 |

## 4. 总体分层架构

建议采用 8 层架构：

```text
教材体系层
-> 知识图谱层
-> 内容资产层
-> 内容工厂层
-> 学习编排层
-> 前端体验层
-> 学习数据层
-> 后台与分析层
```

| 层级 | 负责什么 | 关键产物 |
| --- | --- | --- |
| 教材体系层 | 管教材版本、年级、学期、单元、课时 | Textbook、Grade、Semester、Unit、Lesson |
| 知识图谱层 | 管知识点、能力点、前置关系、易错点 | KnowledgePoint、Skill、Misconception、Prerequisite |
| 内容资产层 | 管题目、解析、提示、知识卡、线索、Boss、勋章 | Question、Explanation、KnowledgeCard、Clue、BossTask、Badge |
| 内容工厂层 | 管模板、生产、审核、版本、发布 | Template、ReviewFlow、ContentBatch、PublishVersion |
| 学习编排层 | 把知识路径编排成案件、关卡、Boss 和复习路径 | Case、Level、Route、UnlockRule、ReviewPlan |
| 前端体验层 | 承载孩子可见的学习游戏体验 | Map、CasePage、QuestionPage、WrongCaseHall、ClueLibrary |
| 学习数据层 | 记录答题、错题、掌握状态、复习和进度 | AnswerRecord、WrongRecord、MasteryState、Progress |
| 后台与分析层 | 支撑教研配置、内容审核、数据看板和发布 | CMS、Analytics、Release、AuditLog |

## 5. 教材体系层

### 5.1 为什么必须独立

后续要支持教材切换，必须把教材作为独立维度，而不是写在题目字符串或关卡名称里。

例如同样是“除法”，不同教材可能：

- 单元顺序不同。
- 知识点拆分不同。
- 例题情境不同。
- 难度推进不同。
- 课时边界不同。

因此底层必须先抽象教材体系。

### 5.2 核心对象

```text
Subject
TextbookVersion
Grade
Semester
Unit
Lesson
CurriculumNode
```

建议层级：

```text
Subject 学科
-> TextbookVersion 教材版本
  -> Grade 年级
    -> Semester 学期
      -> Unit 单元
        -> Lesson 课时 / 主题
          -> CurriculumNode 教材节点
```

### 5.3 教材切换规则

切换教材时，系统应该替换：

- 教材目录。
- 单元结构。
- 知识点映射。
- 题目模板和题目资产。
- 知识卡内容。
- 案件主题和关卡编排。

系统不应该替换：

- 答题引擎。
- 错题悬案馆。
- 线索库机制。
- Boss 解锁机制。
- 勋章系统基础能力。
- 学习数据结构主干。

### 5.4 教材映射示例

```text
subject: math
textbookVersion: bsd
grade: 2
semester: 2
unit: division-unit-1
lesson: division-with-remainder
curriculumNode: remainder-rule
```

如果切换到另一套教材，应该变成新的教材节点，但可以映射到同一个通用知识点：

```text
subject: math
textbookVersion: rjb
grade: 2
semester: 2
unit: table-division
lesson: remainder
curriculumNode: remainder-rule
mapsToKnowledgePoint: math.division.remainder-rule
```

## 6. 知识图谱层

### 6.1 知识体系结构

教材节点解决“书上怎么排”，知识图谱解决“孩子到底要掌握什么”。

建议结构：

```text
KnowledgeDomain 知识领域
-> KnowledgePoint 知识点
  -> Skill 能力点
    -> Misconception 易错点
    -> QuestionPattern 题型模式
    -> RemediationStrategy 补救策略
```

### 6.2 核心对象

| 对象 | 说明 |
| --- | --- |
| KnowledgeDomain | 知识领域，如数与代数、图形与几何 |
| KnowledgePoint | 知识点，如有余数除法 |
| Skill | 能力点，如理解余数含义、判断余数必须小于除数 |
| Prerequisite | 前置知识，如平均分、乘法口诀 |
| Misconception | 易错点，如余数大于除数仍认为合理 |
| DifficultyBand | 难度层级，如基础、巩固、应用、综合 |
| QuestionPattern | 题型模式，如填空、判断、应用题 |
| RemediationStrategy | 补救策略，如回看知识卡、做同类基础题 |

### 6.3 知识点字段建议

```text
knowledgePointId
subjectId
domainId
name
shortName
definition
learningGoal
gradeBand
prerequisiteIds
skillIds
misconceptionIds
questionPatternIds
knowledgeCardTemplateId
reviewStrategy
status
version
```

### 6.4 知识图谱关系

必须维护以下关系：

| 关系 | 用途 |
| --- | --- |
| 教材节点 -> 知识点 | 支持教材切换和教材同步 |
| 知识点 -> 能力点 | 支持细粒度诊断 |
| 能力点 -> 题型模板 | 支持内容工厂批量生产 |
| 能力点 -> 易错点 | 支持错题归因 |
| 易错点 -> 补救策略 | 支持复习推荐 |
| 知识点 -> 知识卡 | 支持线索库复习 |
| 多知识点 -> Boss | 支持综合挑战 |

### 6.5 除法知识图谱示例

```text
math.division
-> average-sharing
   -> skill.identify-equal-groups
   -> skill.calculate-each-share
-> division-expression
   -> skill.map-story-to-expression
   -> skill.explain-dividend-divisor-quotient
-> division-with-remainder
   -> skill.calculate-quotient-and-remainder
   -> skill.explain-leftover
-> remainder-rule
   -> skill.check-remainder-less-than-divisor
   -> misconception.remainder-equal-divisor
   -> misconception.remainder-greater-than-divisor
-> division-word-problem
   -> skill.choose-quotient-or-remainder-by-context
```

## 7. 内容资产层

### 7.1 内容资产不是只有题目

一个知识点至少要产出以下资产：

| 资产 | 作用 |
| --- | --- |
| 知识点定义 | 给教研和系统使用 |
| 学习目标 | 定义孩子要掌握什么 |
| 能力点 | 支撑诊断和分层 |
| 易错点 | 支撑错题归因 |
| 题型模板 | 支撑题目生产 |
| 基础题 | 支撑普通关卡 |
| 应用题 | 支撑真实情境练习 |
| Boss 综合题 | 支撑多个知识点综合验证 |
| 解析模板 | 支撑统一讲解 |
| 错题提示 | 支撑低压力反馈 |
| 知识卡 | 支撑线索库复习 |
| 线索文本 | 支撑案件推进 |
| 复习策略 | 支撑错题重练 |

### 7.2 内容资产核心对象

```text
Question
QuestionTemplate
Explanation
Hint
KnowledgeCard
Clue
BossTask
Badge
StorySnippet
ReviewTask
```

### 7.3 题目必须绑定的维度

每道题必须绑定：

```text
subjectId
textbookVersionId
gradeId
semesterId
unitId
lessonId
knowledgePointId
skillId
misconceptionId
questionPatternId
difficultyBand
caseId
levelId
```

其中 `caseId` 和 `levelId` 是游戏编排维度，不应该替代教材和知识维度。

### 7.4 内容资产版本

题目、知识卡、线索和 Boss 都需要版本。

原因：

- 教研可能修订答案或解析。
- 不同教材版本内容不同。
- 上线后需要知道孩子答的是哪一版题。
- 历史学习记录需要可追溯。

建议所有内容资产包含：

```text
contentId
contentVersion
status: draft / review / approved / published / archived
createdBy
reviewedBy
publishedAt
source
```

## 8. 内容工厂层

### 8.1 内容生产流水线

内容工厂不是“直接写题”，而是从教材和知识图谱开始生产。

标准流程：

```text
选择教材单元
-> 拆解教材节点
-> 映射知识点
-> 拆解能力点
-> 标记前置知识
-> 整理易错点
-> 设计题型模板
-> 编写题目与解析
-> 编写知识卡
-> 编写线索文本
-> 设计 Boss 综合任务
-> 组装案件和关卡
-> 教研审核
-> 儿童可读性审核
-> 发布到内容版本
```

### 8.2 工厂输入与输出

| 阶段 | 输入 | 输出 |
| --- | --- | --- |
| 教材拆解 | 教材目录、课本内容 | 教材节点 |
| 知识建模 | 教材节点、教研经验 | 知识点、能力点、易错点 |
| 模板设计 | 知识点、能力点 | 题型模板、解析模板 |
| 内容生产 | 模板、案件主题 | 题目、知识卡、线索、Boss |
| 编排组装 | 内容资产、游戏规则 | 案件、关卡、解锁路径 |
| 审核发布 | 内容包 | 发布版本 |

### 8.3 角色分工

| 角色 | 负责内容 |
| --- | --- |
| 教研负责人 | 知识点拆解、能力点、易错点、答案和解析审核 |
| 内容策划 | 案件主题、线索文本、剧情片段、儿童化表达 |
| 产品负责人 | 版本目标、范围、不做清单、体验验收 |
| 技术负责人 | Schema、内容配置、发布和兼容 |
| UI/UX | 页面呈现、儿童可读性、交互状态 |
| 测试 | 流程、内容、数据和兼容性验证 |

### 8.4 审核状态机

```text
draft 草稿
-> content_review 内容审核
-> teaching_review 教研审核
-> child_readability_review 儿童可读性审核
-> approved 已通过
-> published 已发布
-> archived 已归档
```

任何 AI 辅助生成内容都只能进入 `draft`，不能跳过人工审核直接发布。

## 9. 学习编排层

### 9.1 编排层的职责

学习编排层负责把知识路径包装为游戏路径。

它要解决：

- 哪些知识点组成一个案件。
- 哪些能力点组成一个关卡。
- 哪些题目进入普通练习。
- 哪些线索在何时发放。
- 哪些知识点解锁 Boss。
- 错题如何进入悬案馆。
- 复习如何回到知识点和能力点。

### 9.2 编排对象

```text
Case
CaseEpisode
Level
LevelRoute
UnlockRule
ClueRewardRule
BossRule
ReviewRoute
BadgeRule
```

### 9.3 从知识到案件的映射

```text
Unit 单元
-> Case 案件
KnowledgePoint 知识点
-> Level 关卡
Skill 能力点
-> QuestionGroup 题组
Misconception 易错点
-> WrongCategory 错题分类
KnowledgeCard 知识卡
-> ClueLibrary 线索库
Multiple KnowledgePoints 多知识点
-> BossTask 综合挑战
```

### 9.4 Boss 不应只是最后两道题

Boss 应该是综合验证任务。

Boss 组装规则建议：

```text
选择本案核心知识点
-> 选择每个知识点的关键能力点
-> 选择常见易错点
-> 设计综合情境
-> 生成多步骤题或题组
-> 配置分步解析
-> 配置失败后的复习路径
```

Boss 失败后系统应该知道：

- 哪个知识点没掌握。
- 哪个能力点有问题。
- 可能对应哪个易错点。
- 推荐回看哪张知识卡。
- 推荐重做哪类题。

## 10. 学习数据层

### 10.1 数据不只是记录分数

学习数据的目标是诊断掌握状态，而不是只显示对错。

基础数据流：

```text
用户作答
-> AnswerRecord
-> SkillPerformance
-> WrongRecord
-> MasteryState
-> ReviewPlan
-> ParentReport / Analytics
```

### 10.2 核心数据对象

```text
User
LearnerProfile
Progress
AnswerRecord
WrongRecord
SkillPerformance
KnowledgeMastery
ReviewRecord
BossAttempt
BadgeRecord
LearningSession
```

### 10.3 答题记录必须保留的映射

每条答题记录至少要保留：

```text
userId
sessionId
questionId
questionVersion
caseId
levelId
subjectId
textbookVersionId
unitId
knowledgePointId
skillId
misconceptionId
userAnswer
correctAnswer
isCorrect
attemptIndex
answeredAt
timeSpent
```

这样后续才能按教材、单元、知识点、能力点和错因做诊断。

### 10.4 掌握状态建议

知识掌握不建议只用“已掌握 / 未掌握”，可以分层：

```text
not_started 未开始
learning 学习中
unstable 不稳定
mastered 已掌握
needs_review 需要复习
```

计算依据可以包括：

- 最近答题正确率。
- 错误次数。
- 是否完成对应知识卡。
- 是否通过 Boss 相关题。
- 是否完成错题复习。

## 11. 前端体验层

### 11.1 前端不直接理解教材

前端应该消费编排后的学习任务，而不是直接写死教材逻辑。

前端拿到的是：

```text
当前案件
当前关卡
当前题组
当前线索
当前知识卡
当前错题
当前 Boss
当前奖励
```

背后这些内容已经由教材体系、知识图谱和编排层决定。

### 11.2 前端核心模块

```text
Start
Map
CaseDetail
LevelPlayer
QuestionPlayer
ClueReward
BossPlayer
CaseClosing
WrongCaseHall
ClueLibrary
BadgeHall
ParentSummary
```

### 11.3 MVP 到正式工程的迁移原则

MVP 可以继续是静态单页，但正式工程应逐步拆为：

```text
content config
game engine
state store
ui components
data adapter
analytics adapter
```

不要让题目、关卡、知识点和游戏状态全部混在一个脚本里。

## 12. 后台与分析层

### 12.1 内容后台长期能力

V3.0 之前可以先用结构化文件管理内容。平台化阶段需要内容后台。

后台能力包括：

- 教材目录管理。
- 知识点地图管理。
- 题型模板管理。
- 题库管理。
- 知识卡管理。
- 案件和关卡编排。
- Boss 配置。
- 勋章配置。
- 内容审核流。
- 内容版本发布。
- 数据看板。

### 12.2 后台角色权限

| 角色 | 权限 |
| --- | --- |
| 内容编辑 | 创建题目、知识卡、线索和剧情 |
| 教研审核 | 审核知识点、答案、解析、难度 |
| 产品负责人 | 配置版本范围、案件结构、上线计划 |
| 设计负责人 | 审核页面呈现和儿童可读性 |
| 技术负责人 | 管理 Schema、发布、回滚和兼容 |
| 数据分析 | 查看学习指标和内容效果 |

## 13. 推荐工程架构

### 13.1 阶段化工程路线

| 阶段 | 工程形态 | 说明 |
| --- | --- | --- |
| v0.x | 静态单页 + 本地内容配置 | 快速验证闭环 |
| v1.x | 前端工程化 + 结构化内容文件 | 支持多案件、多题库、更清晰状态管理 |
| v2.x | 前端 + 后端 API + 数据库 | 支持用户进度、跨设备、家长反馈 |
| v3.x | 内容后台 + 审核流 + 发布系统 | 支持非技术人员配置内容 |

### 13.2 推荐代码分层

```text
apps/
  web/                 # 学生端体验
  admin/               # 后台，后续阶段
packages/
  content-schema/      # 教材、知识点、题目、案件 Schema
  content-runtime/     # 内容读取、校验、版本适配
  game-core/           # 解锁、判分、进度、奖励、复习策略
  ui/                  # 通用 UI 组件
content/
  textbooks/           # 教材目录
  knowledge/           # 知识图谱
  cases/               # 案件编排
  questions/           # 题库
  cards/               # 知识卡
  badges/              # 勋章
docs/
scripts/
```

### 13.3 核心包职责

| 包 | 职责 |
| --- | --- |
| content-schema | 定义所有内容对象的类型和校验规则 |
| content-runtime | 加载内容包、做版本兼容、提供查询接口 |
| game-core | 处理学习进度、解锁、Boss、错题、勋章 |
| ui | 渲染按钮、题目卡、线索卡、悬案卡、勋章等 |
| web | 学生端页面、路由、状态和交互 |
| admin | 内容配置、审核、发布和看板 |

## 14. 内容包设计

### 14.1 内容包是什么

内容包是某个教材、年级、单元或案件的结构化内容集合。

一个内容包应包括：

```text
manifest
textbook nodes
knowledge map
case route
levels
questions
```

### 14.3 内容包切换

切换教材或单元时，应用只需要选择不同内容包：

```text
subject=math
textbook=bsd
grade=2
semester=2
unit=unit-1-division
```

或：

```text
subject=math
textbook=rjb
grade=2
semester=2
unit=unit-division
```

前端和游戏核心不应该关心教材文件怎么组织，只关心内容运行时给出的当前学习任务。

## 15. MVP 在整体架构中的位置

当前 MVP 是 Phase 0 的最小实现，不代表最终架构。

MVP 可以简化为：

```text
静态内容数据
-> 简单线性关卡
-> 本地答题和错题记录
-> 本地线索和勋章
```

但从 v0.2.0 开始，建议逐步做以下拆分：

1. 把题目、关卡、知识卡、勋章从 JS 逻辑中抽到内容配置。
2. 给题目补齐教材、知识点、能力点、错因字段。
3. 增加内容包 manifest。
4. 增加内容校验脚本。
5. 把解锁、判分、错题、勋章规则抽成 game-core。
6. 为后续后端和后台预留数据接口。

## 16. 架构阶段目标

### 16.1 v0.2.0 架构目标

- 形成项目整体架构文档。
- 形成内容 Schema 文档。
- 形成第一案内容包样例。
- 静态 MVP 仍可运行。
- 内容和逻辑开始分离。

### 16.2 v1.0 架构目标

- 学生端前端工程化。
- 内容以结构化文件加载。
- 支持一个正式单元的完整内容包。
- 答题、错题、线索、勋章逻辑模块化。
- 本地数据结构有版本号和迁移策略。

### 16.3 v2.0 架构目标

- 支持多个数学单元。
- 支持跨单元错题和知识卡。
- 支持用户进度更稳定保存。
- 可接入后端数据服务。

### 16.4 v3.0 架构目标

- 支持内容管理后台。
- 支持多教材版本和多学科内容。
- 支持教研审核流和发布流。
- 支持内容效果数据分析。

## 17. 当前关键架构决策

当前建议冻结以下决策：

```text
1. 教材体系、知识图谱、内容资产、学习编排必须分层。
2. 知识点不能只作为题目字段，必须成为可维护的核心对象。
3. 案件和关卡是知识路径的游戏化编排，不是知识体系本身。
4. 题目必须绑定教材节点、知识点、能力点和错因。
5. Boss 必须由多个知识点和能力点组装，不能只是固定最后题。
6. 错题记录必须能回溯到知识点、能力点和易错点。
7. MVP 可以静态配置，但从 v0.2.0 起必须开始内容和逻辑分离。
8. AI 可以辅助内容生产，但不能绕过人工审核。
```
