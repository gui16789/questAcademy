## 2. 背景

v1.1.0 已完成内容包 registry、按内容包隔离的学习记录、内容校验和浏览器回归能力。下一阶段不应直接堆新题，而应先把“可切换内容包”从技术能力推进为受控产品能力，并在新增第二个内容包前执行内容工厂审核门禁。

产品概念层必须保持以下关系：

```text
课程：语文、数学、英语
地点：课程地图上的教材单元节点
内容包：某个地点可运行的案件、题目、线索和知识卡集合
```

因此，`contentPackageId` 仍然是运行时加载和进度隔离的技术单位，但前端不应把单元内容包直接称为“课程”。当前数学课程地图下已有两个地点节点：第一单元“除法”和第二单元“方向与位置”。

## 3. 版本目标

```text
把 v1.1.0 的内容扩展底座升级为可试用的“课程地图 / 单元地点”入口，并完成第二内容包接入前的生产、审核和回归闭环。
```

## 4. 本次包含

- 轻量课程地图 / 单元地点入口，读取 `content/registry.json`。
- 当前内容包标题、案件名和启动页文案从 runtime / registry 派生，不写死除法单元。
- 第二内容包接入前的选题、交付清单和审核记录。
- 第二内容包 draft 接入验证，可通过 `?contentPackageId=...` 和地点入口加载。
- `validate-content --all` 和 `test:regression --all` 覆盖所有 registry 内容包。
- 移动端 320px / 390px 课程地图 / 单元地点入口和答题主流程无横向溢出。

## 5. 本次不包含

- 不做复杂课程中心。
- 不做账号系统、后端、CMS 或家长端。
- 不批量新增多个内容包。
- 不把未审核内容包标记为 `published`。
- 不改变 game-core 的核心通关、错题、勋章规则。
- 不新增强竞技、排行榜、商城或高压考试表达。

## 6. 关键原则

| 编号 | 原则 | 说明 |
| --- | --- | --- |
| P1 | 文档先行 | 版本目标、范围、不做清单和验收标准先冻结 |
| P2 | 分支开发 | 不直接在 `main` 上开发，使用独立分支和 PR |
| P3 | 审核先于发布 | 第二内容包进入 registry 前必须有审核记录 |
| P4 | 内容包独立版本 | 产品版本和内容包版本继续独立 |
| P5 | 入口轻量 | 课程地图 / 单元地点切换是当前学习入口的一部分，不做复杂课程中心 |
| P6 | 体验不退化 | 默认除法内容包 4 关 + Boss + 结案闭环必须保持 |

## 7. 第二内容包接入前门禁

第二内容包进入 registry 前，至少需要完成：

- 明确教材版本、年级、学期、单元和页码来源。
- 完成 `textbook.json`、`knowledge-map.json`、`cases/*.json`、`questions/*.json`、`cards/knowledge-cards.json`、`clues.json`、`badges.json`、`review-rules.json`、`manifest.json`。
- 在 `docs/reviews/` 下新增对应审核记录。
- 内容审核、教研审核、儿童可读性审核无 P0 / P1 阻塞。
- `npm run validate:content -- --package {contentPackageId}` 通过。
- `npm run test:regression -- --package {contentPackageId}` 通过。

## 8. 验收标准

- `npm run check` 通过。
- `npm run test` 通过。
- `npm run validate:content` 通过。
- `npm run validate:content -- --all` 通过。
- `npm run build` 通过。
- `npm run test:regression` 通过。
- `npm run test:regression -- --all` 通过。
- 默认内容包不受影响，刷新后学习记录不丢失。
- 地点切换后学习记录按 `questAcademy:progress:{contentPackageId}` 隔离。
- 320px 移动端无横向溢出，主要按钮不被遮挡。

## 9. 发布停点

gh release create v1.2.0
