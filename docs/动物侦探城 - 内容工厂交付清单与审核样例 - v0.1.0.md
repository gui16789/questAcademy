## 2. 文档目的

本文档把内容工厂的交付物、审核模板、样例记录和人工门禁整理成可执行清单。

任何新内容包进入 registry 或发布前，都应先完成本文档中的交付清单，并留下内容审核、教研审核、儿童可读性审核和技术检查记录。

## 3. 使用方式

建议每个内容包复制本文档第 6-10 节，单独形成一份审核记录，例如：

```text
docs/reviews/{contentPackageId}@{contentVersion}-review.md
```

v1.1.0 暂不要求建立 `docs/reviews/` 目录；正式新增第二个内容包时再落地该目录。

## 5. 人工门禁

内容包状态只允许按以下顺序推进：

```text
draft
-> content_review
-> teaching_review
-> readability_review
-> approved
-> published
-> archived
```

| 状态 | 进入条件 | 退出条件 | 阻塞规则 |
| --- | --- | --- | --- |
| draft | 内容初稿开始生产 | 交付清单基础资产齐全 | 缺少 manifest 或核心题组时不得送审 |
| content_review | 内容负责人提交审核 | 内容完整性和文案一致性 approved | 剧情冲突、字段缺失、游戏化表达越界时退回 draft |
| teaching_review | 内容审核通过 | 教材一致、答案、解析、难度 approved | 答案错误、知识点错误、解析误导为 P0，必须退回 |
| readability_review | 教研审核通过 | 儿童可读性 approved | 低年级无法独立理解、答错压力过高时退回 |
| approved | 三类人工审核和技术检查均通过 | 发布负责人确认进入 published | 任一 P0/P1 未关闭不得 approved |
| published | 产品版本或内容包发布确认 | 后续修订进入新内容包版本 | 发布后发现 P0 需 hotfix 或回滚 |
| archived | 内容包停止使用 | 保留学习记录追溯能力 | 不删除历史学习记录引用 |

发布前必须同时满足：

```text
内容审核 = approved
教研审核 = approved
儿童可读性审核 = approved
技术 Schema 检查 = approved
无 P0 / P1 未关闭问题
validate-content 通过
regression smoke 通过
registry / manifest status 已确认
```

## 10. 问题记录模板

| 编号 | 对象类型 | 对象 ID | 问题类型 | 严重程度 | 问题描述 | 修改建议 | 负责人 | 状态 |

| QA-001 | question |  | answer / explanation / wording / schema | P0 / P1 / P2 / P3 |  |  |  | open |

严重程度定义：

| 等级 | 定义 | 发布影响 |
| --- | --- | --- |
| P0 | 答案错误、知识点错误、教材严重不一致 | 阻塞 approved / published |
| P1 | 解析误导、字段缺失、关键引用错误、儿童明显无法理解 | 阻塞 approved / published |
| P2 | 文案不够清楚、剧情衔接问题、非关键难度偏差 | 可修复后复审 |
| P3 | 表达润色、格式优化、轻微一致性问题 | 不阻塞，但建议修复 |
