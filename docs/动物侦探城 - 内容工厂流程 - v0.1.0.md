## 2. 文档目的

本文档定义《动物侦探城》的内容工厂流程，用于规范教材、知识点、题目、知识卡、线索、Boss、案件和复习规则的生产、审核、发布和迭代。

内容工厂的目标不是“多写题”，而是建立一套可以长期复用、可审核、可扩展、可数据迭代的生产体系。

## 3. 已确认关键决策

| 编号 | 决策 | 说明 |
| --- | --- | --- |
| D1 | 内容工厂采用教材到数据反馈的完整流水线 | 从教材输入到发布后数据回收 |
| D2 | 最小生产单元是 KnowledgePoint Package 知识点内容包 | 不是单道题 |
| D3 | Boss 必须由多个知识点内容包组合生成 | Boss 是综合验证任务 |
| D4 | 审核采用内容审核、教研审核、儿童可读性审核三段 | 保证完整性、准确性和儿童友好 |
| D5 | AI 只能辅助产出 draft | 不能绕过人工审核 |
| D6 | 数据回收纳入内容工厂闭环 | 用真实学习数据反向优化内容 |

## 4. 内容工厂总原则

### 4.1 知识先于剧情

```text
先确定孩子要掌握什么，再决定用什么案件包装。
```

案件、线索、角色和剧情必须服务知识点。不能为了剧情有趣而牺牲教材一致性、知识准确性和题目清晰度。

### 4.2 内容包先于页面

内容生产应先形成结构化内容包，再进入页面和交互实现。

页面不直接创造知识结构，页面只呈现内容包和学习编排结果。

### 4.3 审核先于发布

任何题目、解析、知识卡、Boss 和复习规则都必须经过人工审核后才能发布。

AI 可以帮助起草，但不能替代教研审核和最终确认。

### 4.4 数据反向迭代

内容上线不是结束。上线后的错题率、Boss 通过率、知识卡查看率、错题侦破率和用户反馈，必须反向驱动内容修改。

## 5. 内容工厂主流水线

标准流程：

```text
教材输入
-> 教材节点拆解
-> 通用知识点映射
-> 能力点拆解
-> 易错点拆解
-> 题型模板设计
-> 题目生产
-> 知识卡生产
-> 线索与剧情包装
-> Boss 综合任务设计
-> 案件与关卡编排
-> 内容审核
-> 教研审核
-> 儿童可读性审核
-> 内容包发布
-> 数据回收与迭代
```

每个环节必须有输入、输出、负责人和验收标准。

## 7. 最小生产单元：KnowledgePoint Package

### 7.1 定义

内容工厂的最小生产单元是：

```text
KnowledgePoint Package 知识点内容包
```

不是单道题，也不是单个关卡。

一个知识点内容包表示某个知识点在某个教材单元下可被教学、练习、复习和诊断的完整内容集合。

### 7.2 必备内容

每个知识点内容包至少包含：

| 内容 | 用途 |
| --- | --- |
| 知识点定义 | 说明这个知识点是什么 |
| 学习目标 | 说明孩子要掌握什么 |
| 教材节点映射 | 说明来自哪本教材、哪个单元、哪个课时 |
| 前置知识 | 说明学习前需要什么基础 |
| 能力点 | 支持细粒度诊断 |
| 易错点 | 支持错题归因 |
| 题型模板 | 支持批量生产题目 |
| 基础题 | 支持普通关卡 |
| 巩固题 | 支持练习和复习 |
| 应用题 | 支持生活情境和 Boss |
| 错题提示 | 支持低压力反馈 |
| 解析模板 | 支持统一讲解 |
| 知识卡 | 支持线索库复习 |
| 线索文本 | 支持案件推进 |
| 复习规则 | 支持错题补救 |

## 8. 教材节点拆解

### 8.1 输入

- 教材版本。
- 年级。
- 学期。
- 单元目录。
- 课本页码或课时主题。
- 教师用书或教研参考。

### 8.2 输出

```text
Subject
TextbookVersion
Grade
Semester
Unit
Lesson
CurriculumNode
```

### 8.3 验收标准

- 单元结构清楚。
- 课时或主题边界清楚。
- 每个教材节点有唯一 ID。
- 教材节点能映射到通用知识点。
- 不把游戏关卡当成教材节点。

## 9. 通用知识点映射

### 9.1 目标

建立教材节点与通用知识点之间的映射，支持多教材版本复用同一套知识图谱。

### 9.2 映射规则

```text
一个教材节点可以映射到一个或多个通用知识点。
一个通用知识点可以被多个教材版本引用。
```

### 9.3 示例

```text
北师大版 二年级下册 第一单元 余数规则
-> math.division.remainder-rule

人教版 二年级下册 有余数除法
-> math.division.remainder-rule
```

### 9.4 验收标准

- 映射关系明确。
- 知识点命名稳定。
- 不因换教材而改变学习数据主干。
- 题目和错题能回到通用知识点。

## 10. 能力点与易错点拆解

### 10.1 能力点拆解

能力点是孩子需要实际表现出来的可观察能力。

示例：

```text
知识点：余数小于除数
能力点 1：能比较余数和除数大小
能力点 2：能判断有余数除法结果是否合理
能力点 3：能修正余数过大的错误结果
```

### 10.2 易错点拆解

易错点用于解释孩子为什么错。

示例：

```text
易错点 1：认为余数可以等于除数
易错点 2：认为余数可以大于除数
易错点 3：不会继续多分一组
```

### 10.3 验收标准

- 每个知识点至少 1 个能力点。
- 每个核心能力点至少 1 个常见易错点。
- 易错点能被题目和错题记录引用。
- 易错点能关联补救策略。

## 11. 题型模板设计

### 11.1 作用

题型模板用于保证题目生产稳定，而不是每道题完全靠临时创作。

### 11.2 模板字段

```text
题型模板 ID：
关联知识点：
关联能力点：
关联易错点：
题型：
难度：
题干结构：
变量范围：
答案规则：
解析规则：
错误提示规则：
适用场景：
```

### 11.3 示例

```text
模板：判断余数是否合理
知识点：余数小于除数
能力点：比较余数和除数大小
易错点：认为余数可以大于除数
题型：判断题 / 单选题
题干结构：{被除数} ÷ {除数} = {商} 余 {余数}，这个算式合理吗？
答案规则：余数 < 除数 为合理，否则不合理
解析规则：先比较余数和除数，再判断还能不能继续分
```

## 12. 题目生产

### 12.1 题目类型

当前支持：

```text
single 单选
judge 判断
fill 填空
```

后续可扩展：

```text
multi 多选
match 连线
drag 拖拽
step 多步骤
```

### 12.2 题目生产规则

每道题必须包含：

- 教材维度。
- 知识点 ID。
- 能力点 ID。
- 易错点 ID。
- 题型模板 ID。
- 难度层级。
- 正确答案。
- 解析。
- 错题提示。
- 内容版本。
- 审核状态。

### 12.3 题目数量建议

每个知识点内容包建议至少生产：

| 类型 | 建议数量 |
| --- | --- |
| 基础题 | 4-8 道 |
| 巩固题 | 4-8 道 |
| 应用题 | 2-4 道 |
| 备用题 | 2-4 道 |
| Boss 候选题 | 1-2 道 |

MVP 阶段可降低数量，但字段结构不能省略。

## 13. 知识卡生产

### 13.1 作用

知识卡是孩子复习知识点的主要载体，也是线索库的内容核心。

### 13.2 知识卡结构

```text
知识卡 ID：
关联知识点：
关联能力点：
标题：
一句话解释：
例题：
解题方法：
常见错误：
复习提示：
关联题型模板：
状态：
```

### 13.3 语言要求

- 短句。
- 具体。
- 避免抽象术语。
- 每张卡只讲一个核心知识点。
- 不把剧情线索和知识解释混成一大段。

## 14. 线索与剧情包装

### 14.1 线索的作用

线索是知识点完成后的游戏化反馈，不是独立于学习内容的装饰。

每条线索必须关联：

- 案件。
- 关卡。
- 知识点。
- 知识卡。
- 解锁条件。

### 14.2 线索文本规则

线索文本应该：

- 推动案件。
- 暗含知识点。
- 简短清楚。
- 不替代知识卡解释。

示例：

```text
狐狸配送员说的余数太大，不符合除法规则。
```

### 14.3 剧情包装规则

剧情负责提高参与感，但不能增加理解负担。

原则：

- 每关剧情只交代一个任务。
- 不在题目前放大量故事文本。
- 不使用恐怖、犯罪、暴力表达。
- 不把数学目标藏得太深。

## 15. Boss 综合任务生产

### 15.1 Boss 定义

Boss 是多个知识点和能力点的综合验证任务。

Boss 不是：

- 普通题目的最后两题。
- 单纯更难的计算题。
- 完全由剧情驱动的谜题。

### 15.2 Boss 生产流程

```text
选择核心知识点
-> 选择关键能力点
-> 选择高频易错点
-> 设计综合情境
-> 编写分步题组
-> 配置分步解析
-> 配置失败诊断
-> 配置复习路径
```

### 15.3 Boss 必备字段

```text
Boss ID：
案件 ID：
综合情境：
目标知识点：
目标能力点：
目标易错点：
步骤列表：
每步题目：
通过规则：
失败诊断：
推荐知识卡：
推荐复习题型：
结案关联：
```

### 15.4 Boss 验收标准

- 覆盖本案核心知识点。
- 有明确分步解析。
- 错误能回溯到能力点和易错点。
- 失败后有复习路径。
- 难度是综合，不是故意刁钻。

## 16. 案件与关卡编排

### 16.1 编排原则

案件和关卡是对知识点路径的游戏化编排。

推荐映射：

```text
单元 -> 案件
知识点 -> 关卡
能力点 -> 题组
易错点 -> 错题分类
知识卡 -> 线索库
多知识点 -> Boss
```

### 16.2 关卡编排字段

```text
关卡 ID：
所属案件：
顺序：
名称：
场景：
关联知识点：
关联能力点：
题组：
通过条件：
线索奖励：
知识卡奖励：
解锁规则：
```

### 16.3 通过条件建议

MVP 可使用：

```text
4 题中答对 3 题通过
```

后续可扩展：

```text
按能力点最低正确数通过
按关键题是否答对通过
允许复习后重新判定通过
```

## 17. 审核流程

### 17.1 三段审核

| 阶段 | 审核重点 | 负责人 |
| --- | --- | --- |
| 内容审核 | 字段完整、剧情一致、文案规范、格式正确 | 内容负责人 |
| 教研审核 | 教材一致、知识准确、答案正确、解析清楚、难度合理 | 教研负责人 |
| 儿童可读性审核 | 短句、低压力、孩子能独立理解、无挫败表达 | 产品 / UI / 儿童体验负责人 |

### 17.2 状态流

```text
draft
-> content_review
-> teaching_review
-> readability_review
-> approved
-> published
-> archived
```

### 17.3 退回规则

任一审核阶段发现问题，退回 `draft` 或上一阶段，并记录：

```text
问题类型：
问题位置：
修改建议：
退回人：
退回时间：
是否阻塞发布：
```

## 18. AI 辅助规则

### 18.1 可使用范围

AI 可以辅助：

- 题目初稿。
- 题目变体。
- 解析草案。
- 错题提示草案。
- 知识卡草案。
- 线索文案草案。
- Boss 情境草案。

### 18.2 禁止范围

AI 不能：

- 未经审核直接发布内容。
- 作为最终答案确认者。
- 作为教材一致性最终判断者。
- 绕过教研审核。
- 自动生成大量题目后直接上线。

### 18.3 AI 内容标记

AI 辅助内容必须保留来源标记：

```text
source: ai_assisted
aiModel:
promptVersion:
humanReviewedBy:
humanReviewedAt:
```

## 19. 内容包发布

### 19.1 发布条件

内容包发布前必须满足：

- manifest 完整。
- 教材节点完整。
- 知识点映射完整。
- 题目字段完整。
- 答案和解析通过教研审核。
- 知识卡通过儿童可读性审核。
- Boss 有失败诊断和复习路径。
- 内容版本号确认。
- 回滚版本可用。

### 19.2 发布版本规则

内容包版本建议独立于产品版本，但要能关联产品版本。

示例：

```text
产品版本：v0.2.0
内容包：math.bsd.g2.s2.unit-1-division@0.1.0
```

### 19.3 回滚规则

如果发现内容错误：

- 题目答案错误：立即下线或修订 Patch 内容包。
- 解析不清楚：进入 Patch 修订。
- 个别文案问题：Patch 修订。
- 知识点映射错误：暂停相关内容发布，重新审核。
- 大面积教材不一致：回滚到上一内容包版本。

## 20. 数据回收与迭代

### 20.1 核心指标

内容工厂需要观察：

| 指标 | 用途 |
| --- | --- |
| 题目正确率分布 | 判断题目难度是否合理 |
| 错题集中度 | 找出高频易错点 |
| Boss 通过率 | 判断综合挑战难度 |
| 知识卡查看率 | 判断线索库是否被使用 |
| 错题侦破率 | 判断复习是否有效 |
| 儿童停顿点 | 判断文案或交互是否不清楚 |
| 家长认可度 | 判断学习价值是否被理解 |
| 教研返工率 | 判断内容生产质量 |

### 20.2 迭代动作

根据数据采取：

- 调整题目难度。
- 增加基础题或巩固题。
- 修改解析。
- 改写知识卡。
- 调整 Boss 步骤。
- 优化线索文案。
- 调整关卡顺序。
- 增加复习规则。

### 20.3 复盘节奏

建议：

```text
MVP 测试后复盘一次
每个内容包发布后 3-7 天复盘一次
每个版本结束后做完整内容复盘
```

## 22. 当前待确认问题
