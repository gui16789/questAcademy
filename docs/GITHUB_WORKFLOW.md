# GitHub 文档与代码流程规范

## 1. 仓库目标

本仓库用于沉淀《动物侦探城》的产品文档、内容资产、UI 规范、静态 MVP 代码和后续正式工程代码。

仓库需要同时服务三类工作：

- 产品与项目规划
- 内容生产与审核
- 前端、后端和数据系统开发

## 3. 文档分层

建议维护以下文档层级：

| 层级 | 文档 | 作用 |
| --- | --- | --- |
| L1 | 产品愿景与边界 | 定义为什么做、给谁做、不做什么 |
| L2 | PRD | 定义当前版本做什么、如何验收 |
| L2 | GDD | 定义世界观、角色、案件、奖励和反馈 |
| L2 | 技术方案与架构 | 定义系统如何实现和扩展 |
| L2 | 内容工厂规范 | 定义题目、线索、知识卡和 Boss 如何生产 |
| L3 | 版本计划与复盘 | 定义里程碑、发布记录和复盘结论 |

## 4. Issue 使用规则

Issue 建议分为：

- `docs`: 文档补充或修订
- `product`: 产品需求或范围讨论
- `content`: 题目、剧情、线索、知识卡、Boss
- `design`: UI、视觉、交互、资产
- `frontend`: 页面和交互实现
- `backend`: API、数据库、用户系统
- `bug`: 缺陷
- `research`: 调研、验证、用户测试

每个 Issue 应尽量包含：

```text
背景：
目标：
范围：
不做：
验收标准：
关联文档：
```

## 5. PR 流程

所有文档、内容和代码变更必须通过 Issue、独立分支和 PR 管理。禁止直接在 `main` 上开发。

标准流程：

```text
创建 Issue
-> 明确范围和验收标准
-> 从 main 拉分支
-> 修改文档 / 内容 / 代码
-> 自测
-> 提交 PR
-> Review
-> 合并 main
```

文档和内容类 PR 可以更小、更频繁；架构和核心流程类 PR 需要更严格评审。

PR 合并前必须确认：

- 已关联 Issue。
- 变更范围与 Issue 一致。
- 已完成自测或说明无法测试的原因。
- 相关文档已同步。
- 无 P0 / P1 问题。

分支命名建议：

```text
docs/<issue-id>-<topic>
feature/<issue-id>-<module>
fix/<issue-id>-<bug>
content/<issue-id>-<case-or-unit>
test/<issue-id>-<scope>
release/<version>
```

## 6. Review 重点

产品类：

- 是否服务当前版本目标
- 是否强化核心学习闭环
- 是否增加范围失控风险

内容类：

- 知识点是否准确
- 题目难度是否适合二年级
- 解析是否能帮助孩子理解
- 错题分类是否明确

设计类：

- 孩子是否知道下一步点哪里
- 题目是否足够清晰
- 移动端是否安全
- 是否遵守图片资产分层规则

技术类：

- 数据结构是否可扩展
- 状态流转是否清晰
- 是否影响已有学习记录
- 是否便于后续内容配置化

## 7. 版本节奏

建议从以下版本开始：

```text
v0.1.0  静态 MVP 资产归档
v0.2.0  MVP 文档冻结与测试准备
v1.0.0  除法正式体验版
```

每次发布应在 GitHub Release 中记录：

```text
新增：

回滚方案：
下一步：
```
