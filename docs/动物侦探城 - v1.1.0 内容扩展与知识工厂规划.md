## 2. 背景

v1.0.0 已完成 React + TypeScript + Vite 工程化，并把内容包、内容 runtime、game-core、学生端 UI 和 localStorage 迁移拆开。

当前系统已经证明：

- 单个内容包可以被工程加载。
- 游戏规则可以脱离页面代码运行。
- 学习记录可以绑定内容包版本。
- 当前除法单元可以完整跑通 4 关 + Boss + 结案。

但 v1.0.0 仍然是“单内容包正式体验版”。如果要支持更多教材、单元和知识点，v1.1.0 需要先解决内容扩展方式，而不是直接堆新题。

## 3. v1.1.0 版本目标

```text
把当前单内容包应用升级为可注册、可切换、可审核、可回归的内容扩展底座。
```

v1.1.0 的核心不是新增大量内容，而是建立下一批内容可以稳定生产和接入的机制。

### 3.1 目标

- 支持前端从内容包 registry 选择默认内容包。
- 支持未来按教材、年级、学期、单元切换内容包。
- 明确内容包生产流水线和审核门槛。
- 明确知识点、能力点、易错点、题型模板的生产关系。
- 明确内容包版本与产品版本的关系。
- 补齐内容工厂的校验、回归和发布规则。

### 3.2 不做

- 不接后端。
- 不做正式 CMS。
- 不做账号系统。
- 不做家长端。
- 不大规模新增内容包。
- 不改变 v1.0.0 的核心游戏体验。
- 不把教材切换做成复杂商业化课程中心。

## 4. 已确认原则

| 编号 | 原则 | 说明 |
| --- | --- | --- |
| P1 | 产品版本与内容包版本独立 | Git tag 跟产品版本一致；内容包单独维护版本 |
| P2 | 内容包是运行时输入 | 前端和 game-core 不应写死某一个单元 |
| P3 | 知识体系先于案件 | 案件从知识路径中编排出来，不反向决定知识结构 |
| P4 | registry 先于多内容 | 先有内容包登记、选择、校验和回归，再新增内容 |
| P5 | 审核先于发布 | AI 可辅助 draft，但不能绕过人工审核 |
| P6 | 学习记录绑定内容包 | 进度、错题、勋章必须能追溯到内容包和内容版本 |

## 5. v1.1.0 目标架构

v1.1.0 后，前端内容加载关系应从：

```text
apps/web
-> bundled current division package
```

演进为：

```text
apps/web
-> content package registry
-> selected content package
-> content-runtime
-> game-core
-> React pages
```

### 5.1 内容包 registry

registry 负责描述“有哪些内容包可以被应用加载”。

建议最小字段：

```text
contentPackageId
contentVersion
title
subjectId
textbookVersionId
gradeId
semesterId
unitId
defaultCaseId
status
entryPath
isDefault
releaseNote
```

v1.1.0 可以先用静态 JSON 或 TS 配置实现，不需要后端。

### 5.2 内容包选择策略

v1.1.0 建议只实现三层策略：

```text
1. 如果 URL 或配置指定 contentPackageId，则加载指定内容包
2. 否则加载 registry 中 isDefault = true 的内容包
3. 如果指定内容包不可用，则回退默认内容包并提示
```

不建议在 v1.1.0 做复杂课程选择页。可以保留一个内部调试入口或简单切换控件。

### 5.3 学习记录 key 策略

当前 v1.0.0 仍使用：

```text
animalDetectiveCityMvp
```

v1.1.0 需要引入内容包维度，避免不同内容包互相覆盖。

建议：

```text
questAcademy:progress:{contentPackageId}
```

兼容策略：

```text
animalDetectiveCityMvp
-> 只作为 v0.2.0 / v1.0.0 旧数据迁移来源
-> 迁移后写入新 key
```

## 6. 教材切换扩展点

教材切换不是只换题目，而是切换一组结构化内容：

| 维度 | 切换时是否变化 | 说明 |
| --- | --- | --- |
| Textbook | 是 | 教材版本、目录、单元结构 |
| KnowledgeMap | 可能 | 通用知识点尽量复用，教材映射变化 |
| Case | 是 | 案件主题可随单元变化 |
| Levels | 是 | 关卡数量、顺序、知识点路径 |
| Questions | 是 | 题目、解析、提示 |
| KnowledgeCards | 是 | 知识卡文本和例题 |
| Clues | 是 | 案件线索文本 |
| Badges | 可复用 | 勋章机制可复用，文案可局部变化 |
| ReviewRules | 是 | 复习规则绑定技能和易错点 |
| game-core | 否 | 判分、解锁、错题、勋章规则尽量稳定 |

v1.1.0 要确保代码层只依赖 content-runtime 输出，不直接依赖某个除法单元。

## 7. 知识工厂生产流水线

v1.1.0 需要把现有“内容工厂流程”收敛成可执行流水线。

标准流水线：

```text
教材输入
-> 教材节点拆解
-> 通用知识点映射
-> 能力点拆解
-> 易错点拆解
-> 题型模板设计
-> 题目生产
-> 知识卡生产
-> 线索与剧情包装
-> Boss 综合任务设计
-> 案件与关卡编排
-> 内容 Schema 校验
-> 内容审核
-> 教研审核
-> 儿童可读性审核
-> 内容包发布
-> 回归测试
-> 数据回收与迭代
```

### 7.1 v1.1.0 最小知识工厂产物

每个新内容包至少要交付：

- `manifest.json`
- `textbook.json`
- `knowledge-map.json`
- `cases/*.json`
- `questions/*.json`
- `cards/knowledge-cards.json`
- `clues.json`
- `badges.json`
- `review-rules.json`
- 内容审核记录
- 内容回归记录

### 7.2 生产状态

建议内容包状态：

```text
draft
content_review
teaching_review
readability_review
approved
published
archived
```

v1.1.0 不需要自动化流转，但文档和校验应承认这些状态。

### 7.3 AI 辅助边界

AI 可以做：

- 教材节点初步拆解。
- 题型模板草拟。
- 题目 draft。
- 知识卡 draft。
- 线索和剧情包装 draft。

AI 不可以做最终确认：

- 知识点准确性。
- 答案和解析正确性。
- 教材一致性。
- 儿童可读性。
- 发布批准。

## 8. 内容校验扩展

v1.0.0 的 `validate-content` 已能校验当前内容包。v1.1.0 需要扩展为“多内容包校验”。

建议：

```bash
npm run validate:content
npm run validate:content -- --package math.bsd.g2.s2.unit-1-division
npm run validate:content -- --all
```

校验范围新增：

- registry 中的内容包 entryPath 必须存在。
- registry 默认内容包只能有一个。
- 每个 registry 内容包都能独立加载。
- 学习记录 key 生成规则稳定。
- 内容包 status 不满足发布条件时不能进入 release。

## 9. 回归测试扩展

v1.0.0 已有：

```bash
npm run test:regression
```

v1.1.0 需要支持：

```bash
npm run test:regression -- --package {contentPackageId}
npm run test:regression -- --all
```

最小验收：

- 默认内容包完整 4 关 + Boss + 结案。
- 指定内容包可加载。
- 320px 移动端可用。
- localStorage 按内容包隔离。
- 旧 key 迁移不破坏新 key。

## 10. v1.1.0 Issue 拆分草案

建议 PR 顺序：

```text
Issue 1：v1.1.0 规划和文档
-> Issue 2：content registry
-> Issue 3：多内容包 loader
-> Issue 4：storage key 按内容包隔离
-> Issue 5：validate-content 支持 registry / all
-> Issue 6：regression smoke 支持 package 参数
-> Issue 7：内容工厂审核模板和样例补齐
-> Issue 8：v1.1.0 发布准备
```

### Issue 1：v1.1.0 规划和文档

目标：

- 完成本规划文档。
- 更新文档中心。
- 冻结 v1.1.0 范围和不做清单。

不做：

- 不改运行时代码。

### Issue 2：content registry

目标：

- 新增内容包 registry。
- 当前除法内容包登记为默认内容包。
- 明确 registry 类型。

验收：

- 默认内容包可从 registry 查询。
- registry 可被 web 和脚本复用。

### Issue 3：多内容包 loader

目标：

- `apps/web` 不再直接写死当前内容包 import map。
- 通过 registry 加载默认内容包。
- 为未来指定内容包预留接口。

验收：

- v1.0.0 当前体验不变。
- 内容仍来自 content-runtime。

### Issue 4：storage key 按内容包隔离

目标：

- 新增 `questAcademy:progress:{contentPackageId}`。
- 旧 key 只作为迁移来源。
- 同一浏览器中不同内容包进度互不覆盖。

验收：

- v1.0.0 旧数据可迁移。
- 默认内容包刷新后不丢进度。

### Issue 5：validate-content 支持 registry / all

目标：

- 校验 registry。
- 支持校验单个内容包和全部内容包。

验收：

- 当前内容包校验通过。
- registry 引用错误能失败。

### Issue 6：regression smoke 支持 package 参数

目标：

- 回归脚本可指定内容包。
- 为多内容包发布门禁做准备。

验收：

- 默认内容包回归通过。
- 指定不存在内容包时明确失败。

### Issue 7：内容工厂审核模板和样例补齐

目标：

- 补齐内容包生产 checklist。
- 补齐审核记录样例。
- 明确 draft -> approved -> published 的人工门禁。

验收：

- 新内容包接入前有清晰交付清单。

### Issue 8：v1.1.0 发布准备

目标：

- 更新 README。
- 更新 Release Notes。
- 确认产品版本和内容包版本。
- 跑完整命令。

验收：

- 准备打 tag，发布前停下来确认。

## 11. 风险和决策点

| 风险 | 说明 | v1.1.0 处理 |
| --- | --- | --- |
| 过早做课程选择页 | 容易拖大范围 | 先做 registry 和内部切换能力 |
| 内容包格式频繁变动 | 会影响旧内容 | Schema 变更必须有迁移说明 |
| 学习记录混在一起 | 多内容包后会互相污染 | v1.1.0 优先做 key 隔离 |
| 内容生产无审核 | 规模化后质量不可控 | 审核模板先行 |
| 回归只测默认包 | 多包后发布风险高 | 回归脚本支持 package 参数 |

## 12. 发布门槛

v1.1.0 发布前必须满足：

- `npm run check` 通过。
- `npm run test` 通过。
- `npm run validate:content` 通过。
- `npm run build` 通过。
- `npm run test:regression` 通过。
- registry 默认内容包可加载。
- 当前除法内容包体验不退化。
- localStorage v1.0.0 数据可保留。
- 旧 key 迁移策略可验证。
- 无 P0 / P1 Bug。
