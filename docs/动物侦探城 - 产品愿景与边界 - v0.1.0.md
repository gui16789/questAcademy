## 3. 产品一句话定位

《动物侦探城》是一款面向小学低年级儿童的探案式学习产品，先从数学教材同步练习切入，通过“动物城市案件 + 知识点闯关 + 线索收集 + 错题悬案馆 + Boss 综合挑战 + 勋章激励”的方式，让孩子在低压力的故事体验中完成知识学习、练习、复习和阶段巩固。

## 4. 产品愿景

长期愿景：

```text
把低年级孩子觉得枯燥的基础学科学习，转化为可持续扩展、可验证效果、可持续生产内容的探案式学习体验。
```

《动物侦探城》希望成为一个长期可扩展的儿童学习世界：

- 每个学科可以拆成知识地图。
- 每个教材单元可以转化为一个或多个案件区域。
- 每个知识点可以转化为关卡、题目、线索卡和错题复习任务。
- 每次错题不是失败，而是一个“待侦破悬案”。
- 每次复习不是重复刷题，而是一次“重新调查”。
- 每个阶段完成后，孩子能获得明确的成长反馈。
- 家长可以理解孩子学了什么、错在哪里、下一步该复习什么。
- 教研和内容团队可以用模板持续生产可审核内容。

产品最终不是一个单次 MVP 游戏，而是一套可持续生产内容、可持续验证学习效果、可持续扩展学科和年级的游戏化学习平台。

## 5. 核心价值

### 5.1 对孩子的价值

- 把数学练习包装成探案任务，降低开始练习的心理阻力。
- 用线索、案件和勋章建立持续完成感。
- 用温和反馈减少答错后的挫败感。
- 让错题复习变成“重新调查”，提高复习意愿。
- 通过知识卡帮助孩子把题目背后的知识点记住。

### 5.2 对家长的价值

- 孩子更愿意主动完成课后练习。
- 练习内容与教材单元对齐，避免纯娱乐化。
- 可以看到孩子错在哪里、哪些知识点需要复习。
- 通过完成率、错题、Boss 表现判断学习效果。
- 不引入过重游戏系统，避免学习目标被奖励和装饰稀释。

### 5.3 对教研与内容团队的价值

- 建立可复用的案件、关卡、题目、线索卡、Boss 和勋章模板。
- 建立内容工厂流程，减少每次扩展都从零设计。
- 每道题、每张知识卡、每个 Boss 都能回溯到教材、单元和知识点。
- 通过人工审核机制保证学习内容准确，不让趣味性压过教育准确性。

### 5.4 对产品与研发团队的价值

- 用数据验证学习闭环是否有效，避免靠主观感觉扩展。
- 技术架构可以从单一 MVP 平滑升级到多单元、多年级、多学科。
- 通过清晰的版本边界控制范围，避免项目在游戏化扩展中失焦。
- 通过结构化内容和组件化系统提高后续迭代效率。

## 6. 目标用户

### 6.1 长期核心用户

| 项目 | 定义 |
| --- | --- |
| 年龄 | 6-10 岁左右 |
| 年级 | 小学低年级为主，后续可扩展到中年级 |
| 学科 | 数学优先，后续可扩展到语文、英语、科学等基础学科 |
| 教材 | 先对齐具体教材版本，后续支持多教材版本 |
| 场景 | 家庭课后练习、家长陪伴或孩子独立使用 |
| 单次时长 | 10-15 分钟 |

### 6.2 首期起始用户

| 项目 | 定义 |
| --- | --- |
| 年龄 | 7-8 岁左右 |
| 年级 | 小学二年级 |
| 学科 | 数学 |
| 教材 | 北师大版小学数学二年级下册 |
| 首期单元 | 第一单元：除法 |

首期从二年级下册数学切入，是为了用一个边界清晰、题型可控、学习目标明确的单元验证核心闭环，而不是把产品长期限定为二年级数学工具。

### 6.3 核心用户特点

- 已具备基础识字能力，但不适合长篇阅读。
- 对动物角色、案件、地图、奖励有天然兴趣。
- 容易因为连续做错题产生挫败感。
- 更适合短流程、强反馈、明确下一步的体验。
- 不适合复杂操作、复杂系统和高压竞技。

### 6.4 重要关联用户

| 用户 | 关注点 |
| --- | --- |
| 家长 | 是否同步教材、是否有学习价值、孩子是否愿意继续、是否能看到错题和掌握情况 |
| 教研人员 | 知识点是否准确、题目难度是否合理、解析是否帮助理解 |
| 内容策划 | 案件是否可持续生产、线索和知识点是否自然结合 |
| 开发与设计 | 系统是否可扩展、组件和数据是否可复用 |

## 7. 项目边界与阶段边界

### 7.1 长期产品边界

《动物侦探城》长期要做的是：

- 面向小学阶段的游戏化学习产品。
- 以教材知识点为基础，而不是脱离教材的纯游戏。
- 以案件、关卡、线索、错题、知识卡、Boss 和勋章为核心体验结构。
- 以内容工厂支撑持续生产，而不是一次性手工堆内容。
- 以学习效果和儿童理解为第一优先级，而不是以游戏时长、付费刺激或竞技排名为第一目标。

长期不把产品定义为：

- 普通刷题网站。
- 成人教育后台。
- 纯剧情游戏。
- 复杂 RPG 养成游戏。
- 高压考试系统。
- 以排名、商城、社交为核心的儿童娱乐社区。

### 7.2 学科与内容边界

长期学科路线：

| 阶段 | 学科边界 | 说明 |
| --- | --- | --- |
| 起步阶段 | 数学 | 先验证题目、错题、知识卡和 Boss 的学习闭环 |
| 扩展阶段 | 数学多单元、多年级 | 建立稳定知识地图和内容工厂 |
| 平台阶段 | 语文、英语、科学等基础学科 | 只有在数学内容工厂稳定后再扩展 |

内容必须满足：

- 对齐明确教材、年级、学期、单元和知识点。
- 每个练习和复习内容可被教研审核。
- 游戏剧情服务知识点，不牺牲学习准确性。
- 错题和知识卡能回到具体知识点。

### 7.3 产品能力边界

长期核心能力包括：

- 学习地图
- 案件系统
- 关卡系统
- 题库系统
- 答题与解析
- 错题悬案馆
- 知识线索库
- Boss 综合挑战
- 勋章与成长反馈
- 学习记录
- 内容管理后台
- 家长反馈
- 教研审核流程

非核心能力必须后置评估：

- 排行榜
- 商城
- 复杂社交
- 多人对战
- 重度养成
- 大量剧情动画
- AI 自动出题直接上线

### 7.4 阶段路线边界

| 阶段 | 目标 | 边界 |
| --- | --- | --- |
| Phase 0 / v0.x | 验证核心学习闭环 | 单学科、单单元、单案件 |
| Phase 1 / v1.0 | 打磨首个正式体验 | 数学二年级下册除法单元 |
| Phase 2 / v1.1 | 扩展同单元内容 | 新增除法案件和题库，不改变核心系统 |
| Phase 3 / v1.5 | 验证跨单元能力 | 扩展到方向与位置等新知识类型 |
| Phase 4 / v2.0 | 形成多单元地图 | 二年级下册多个数学单元 |
| Phase 5 / v2.5 | 增加轻量家长反馈 | 学习报告、错题和复习建议 |
| Phase 6 / v3.0 | 平台化 | 内容后台、多年级、多学科、教研审核工作流 |

每一阶段只解决该阶段最关键的问题，不能提前引入下阶段的大系统。

### 7.5 当前阶段边界

当前阶段是 Phase 0：MVP 验证阶段。

核心任务不是做完整项目，而是验证：

```text
探案式学习闭环是否能让二年级孩子愿意完成数学练习，并愿意回到错题复习和知识点复习。
```

### 7.6 当前内容边界

| 项目 | 范围 |
| --- | --- |
| 教材版本 | 北师大版小学数学二年级下册 |
| 首期单元 | 第一单元：除法 |
| 首个案件 | 徽章失踪案 |
| 普通关卡 | 4 个知识点关卡 |
| Boss | 1 个综合挑战 |
| 题目规模 | 约 28 道题，包括正式题和备用题 |
| 复习系统 | 悬案馆和线索库 |
| 激励系统 | MVP 勋章馆 |

### 7.7 当前产品边界

MVP 必须完成：

- 开始页
- 地图页
- 案件详情页
- 普通关卡答题
- 线索获得
- Boss 挑战
- 结案页
- 悬案馆
- 线索库
- 勋章馆
- 基础学习记录

MVP 不追求：

- 多单元完整地图
- 复杂剧情动画
- 家长端完整报告
- 老师端和班级管理
- 大规模题库
- 长期账号体系
- 内容管理后台

### 7.8 当前技术边界

当前静态 MVP 可以使用本地存储验证学习闭环。

正式工程设计时必须预留：

- 内容配置化能力
- 学习记录持久化能力
- 错题和知识点结构化能力
- 后续用户系统和后端 API 接入能力
- 多单元、多案件、多题型扩展能力

## 8. 明确不做清单

### 8.1 项目级不做

| 功能 | 不做原因 |
| --- | --- |
| 高压考试化 | 与低压力学习体验冲突 |
| 羞辱式错误反馈 | 伤害儿童学习信心 |
| 以消费刺激为核心的商城 | 容易偏离学习目标，并带来儿童合规风险 |
| 未经人工审核的 AI 自动出题直接上线 | 题目质量和教育准确性不可控 |
| 直接使用已有商业 IP | 存在版权和品牌风险 |
| 以排名输赢为核心的儿童竞技系统 | 容易增加比较压力，不符合首要学习目标 |
| 与学习结果无关的重度装饰系统 | 会稀释学习闭环 |

### 8.2 当前 MVP 阶段不做

| 功能 | 不做原因 |
| --- | --- |
| 排行榜 | 不服务首期学习闭环 |
| 商城 | 增加运营和合规成本 |
| 复杂宠物养成 | 玩法成本高，容易稀释学习重点 |
| 多人对战 | 技术和规则复杂，不适合验证单人学习闭环 |
| 复杂社交 | 与当前家庭课后练习场景不匹配 |
| 老师端 | 涉及班级、作业、学生管理，范围过大 |
| 完整家长端 | 当前只需要验证学习闭环，报告可后置 |
| AI 自动出题 | 题目质量和教研审核风险高 |
| 大量动画剧情 | 美术和开发成本高，容易影响交付 |
| 语音识别 | 成本高，不是当前核心验证点 |
| 多学科 | 会稀释除法单元验证重点 |
| 全册内容 | 内容量过大，不利于快速验证 |

### 8.3 需要阶段门后再评估的方向

以下方向不是永久禁止，但必须等核心学习闭环稳定、数据证明有效后再评估：

- 排名和竞技系统
- 高强度付费激励
- AI 辅助内容生成
- 过重剧情系统
- 大量装饰型收集系统
- 与学习结果无关的养成系统
- 老师端与班级管理
- 多学科扩展

## 9. 产品核心闭环

所有版本必须围绕以下闭环设计：

```text
接案
-> 学习知识点
-> 普通关卡练习
-> 收集线索
-> 错题进入悬案馆
-> 知识点进入线索库
-> Boss 综合挑战
-> 结案
-> 获得勋章
-> 复习与再挑战
```

如果一个新功能不能增强这条闭环，它不能优先进入当前版本。

## 10. 产品原则

### 10.1 学习优先

游戏包装必须服务学习目标。

判断标准：

- 题目是否绑定明确知识点。
- 解析是否帮助孩子理解。
- 错题是否能回到知识点。
- Boss 是否考察综合应用，而不是单纯变难。

### 10.2 儿童可理解优先

孩子必须知道：

- 我现在在哪里。
- 我要做什么。
- 下一步点哪里。
- 答错后该怎么办。
- 通关后获得了什么。

### 10.3 低压力反馈

答错不是失败，而是线索还没查清楚。

推荐语言：

```text
待调查线索
未侦破悬案
重新调查
再试一次
成功侦破
```

避免语言：

```text
失败
错误太多
答错了
排名落后
惩罚
```

### 10.4 小步可交付

每个版本都必须能独立体验、独立验收、独立复盘。

不做“大版本憋很久”；优先做能被孩子真实体验、能被数据验证的小版本。

### 10.5 内容模板化

每个新单元必须能沉淀为：

- 知识点地图
- 案件模板
- 关卡模板
- 题目模板
- 线索卡模板
- Boss 模板
- 勋章模板
- 错题分类规则

## 11. IP 与合规原则

### 11.1 IP 原则

《动物侦探城》必须使用原创动物城市设定。

禁止：

- 使用已有商业 IP 的角色名称、形象、地图、剧情。
- 使用容易让用户误认为官方授权的视觉表达。
- 直接复刻已有动画、游戏或绘本角色。

允许：

- 使用通用动物角色概念。
- 使用原创动物城市、侦探学院、徽章、案件设定。
- 使用通用探案元素，如放大镜、档案袋、线索板、徽章。

### 11.2 儿童合规原则

面向低龄儿童时必须遵守：

- 不出现恐怖、暴力、血腥、惊吓内容。
- 不制造羞辱式错误反馈。
- 不使用诱导性消费设计。
- 不做高压竞技和强排名刺激。
- 不收集超出学习体验必要范围的儿童个人信息。
- 若未来引入账号、家长端、支付、AI 或用户上传内容，必须重新评估隐私和合规方案。

### 11.3 数据与隐私原则

项目长期必须坚持最小必要数据原则。

任何阶段都只采集支撑学习体验、学习记录、内容改进和必要账号能力的数据，不采集与学习无关的儿童敏感信息。

当前 MVP 必要数据仅包括：

- 昵称或默认身份
- 学习进度
- 答题记录
- 错题记录
- 线索解锁
- 勋章获得

后续如接入后端和账号系统，必须补充：

- 隐私政策
- 儿童数据保护策略
- 家长授权机制
- 数据删除与导出机制
- 数据安全与权限控制方案

## 12. 成功指标

### 12.1 项目级成功指标

| 指标 | 目标 |
| --- | --- |
| 学习闭环有效 | 孩子能完成学习、练习、复习、综合挑战和阶段总结 |
| 儿童愿意持续使用 | 孩子愿意继续进入下一案或下一单元 |
| 家长认可学习价值 | 家长能理解孩子学了什么、错在哪里、下一步如何复习 |
| 内容生产可复制 | 新单元能按模板生产案件、关卡、题目、线索卡和 Boss |
| 教研质量可控 | 内容上线前有审核流程，题目和解析准确 |
| 技术架构可扩展 | 系统能从单案件扩展到多案件、多单元、多年级 |
| 合规风险可控 | 儿童数据、内容表达、IP 和商业化设计符合低龄产品要求 |

### 12.2 阶段成功指标

| 阶段 | 成功判断 |
| --- | --- |
| Phase 0 / MVP | 核心闭环成立，孩子愿意完成首案并尝试复习 |
| Phase 1 / V1.0 | 首个正式单元体验稳定，内容和数据记录可靠 |
| Phase 2 / V1.1 | 同单元新案件可以复用内容和技术模板 |
| Phase 3 / V1.5 | 新知识类型可以接入同一套系统 |
| Phase 4 / V2.0 | 多单元进度、错题和知识卡可统一管理 |
| Phase 5 / V2.5 | 家长能看懂学习结果和复习建议 |
| Phase 6 / V3.0 | 非技术人员可通过后台配置内容并经过审核上线 |

### 12.3 MVP 成功指标

| 指标 | 目标 |
| --- | --- |
| 案件完成率 | 70% 以上孩子能完成首个案件 |
| 继续体验意愿 | 60% 以上孩子愿意继续玩下一案 |
| 流程理解率 | 70% 以上孩子知道下一步该做什么 |
| Boss 通过率 | 50% - 80% |
| 悬案馆吸引力 | 50% 以上孩子愿意查看或重做错题 |
| 勋章吸引力 | 50% 以上孩子愿意查看勋章 |
| 家长认可度 | 70% 以上家长认为形式有学习价值 |

### 12.4 学习效果观察指标

| 指标 | 观察目的 |
| --- | --- |
| 普通关卡通过率 | 判断知识点和题目难度是否合理 |
| 错题产生率 | 判断题目是否过难或知识点是否薄弱 |
| 错题侦破率 | 判断错题复习机制是否有效 |
| 线索库进入率 | 判断知识卡是否有复习价值 |
| Boss 错题分布 | 判断综合应用是否超出当前学习能力 |

### 12.5 体验观察指标

| 指标 | 观察目的 |
| --- | --- |
| 首次进入案件成功率 | 判断地图和入口是否清楚 |
| 每页停顿点 | 判断孩子在哪些页面困惑 |
| 答错后继续率 | 判断反馈是否足够温和 |
| 结案页完成感 | 判断最终奖励是否有效 |
| 家长旁观干预次数 | 判断孩子能否独立操作 |

## 13. 阶段门规则

### 13.1 通用阶段门

每个阶段结束时必须判断：

- 本阶段的核心学习闭环是否更稳定。
- 目标用户是否能理解主要流程。
- 家长或教研是否认可学习价值。
- 内容生产是否比上一阶段更可复制。
- 技术结构是否能支撑下一阶段范围。
- 当前阶段是否遗留 P0 / P1 风险。
- 下一阶段目标、范围和不做清单是否明确。

如果这些问题不能得到清晰回答，不应进入下一阶段扩展。

### 13.2 Phase 0 进入 V1.0 的条件

同时满足以下条件，才建议从 MVP 验证进入 V1.0 打磨：

- 首个案件核心流程稳定。
- 无 P0 / P1 Bug。
- 孩子大多能理解下一步操作。
- 案件完成率达到 70% 以上。
- Boss 通过率处于 50% - 80%。
- 家长认可学习价值。
- 题目和解析通过教研审核。
- 内容生产流程初步可复制。

### 13.3 不应继续扩展的情况

出现以下情况时，应暂停扩展新单元或新玩法：

- 孩子无法独立理解主流程。
- 错题复习没有使用意愿。
- Boss 难度明显失衡。
- 家长认为游戏化削弱学习价值。
- 内容生产严重依赖临时灵感，无法模板化。
- 当前技术结构无法支撑更多案件。

## 14. 决策规则

当出现产品、内容、设计或技术争议时，按以下顺序判断：

```text
1. 是否影响学习效果？
2. 是否影响儿童理解？
3. 是否影响核心学习闭环？
4. 是否影响当前版本交付？
5. 是否影响后续扩展？
6. 是否只是装饰性或锦上添花？
```

进入当前版本的功能必须满足：

- 服务当前版本目标。
- 增强核心学习闭环。
- 有明确验收标准。
- 不明显增加延期风险。

如果不能同时满足，进入后续版本池。

## 17. 当前冻结结论

作为项目级愿景与边界文档，当前冻结以下长期方向：

```text
产品类型：面向小学低年级的探案式学习产品
学科策略：数学优先，后续扩展到更多基础学科
内容策略：教材知识点驱动，内容必须可审核、可复用、可配置
核心闭环：接案、学习、闯关、线索、错题、知识卡、Boss、结案、勋章、复习
体验原则：儿童可理解、低压力反馈、学习效果优先
扩展原则：先验证闭环，再扩展内容；先稳定单元，再扩展学科和平台化
合规原则：原创 IP、儿童友好、最小必要数据、人工审核内容
```

当前 Phase 0 额外冻结以下阶段边界：

```text
起始学科：数学
起始年级：小学二年级
教材范围：北师大版数学二年级下册第一单元“除法”
首个案件：徽章失踪案
阶段目标：验证探案式学习闭环是否成立
阶段限制：不做多学科、多单元、家长端、老师端、商城、排行榜和复杂养成
```

## 18. 待确认问题

以下问题不阻塞当前文档作为 v0.1.0 草稿使用，但需要在后续版本规划中逐步确认：

1. 长期是否坚持数学为第一主线，还是在 V2.0 前提前探索语文或英语。
2. 多教材版本支持从哪个阶段开始纳入架构设计。
3. 年级扩展顺序是横向扩展同年级多单元，还是纵向扩展相邻年级。
4. 正式版本是否需要账号体系，还是继续以本地游客体验为主。
5. 家长反馈页进入版本是 V1.0、V1.1 还是 V2.5。
6. 老师端和班级管理是否属于 V3.0 平台化范围，还是更后置。
7. 内容审核负责人和审核流程如何落地。
8. 是否需要单独准备儿童隐私与家长授权方案。
9. AI 辅助内容生产可以参与到哪一步，哪些环节必须人工确认。
