## 2. 版本目标

v1.0.0 的目标是把 v0.2.0 静态单页和 JSON 内容包升级为正式前端工程：

```text
React + TypeScript + Vite
+ content-runtime
+ game-core
+ UI 组件化
+ 本地数据迁移
+ 静态部署能力
```

v1.0.0 仍然聚焦首个除法内容包，不扩展新单元、不接后端、不做内容后台。

## 3. 已确认关键决策

| 编号 | 决策 | 说明 |
| --- | --- | --- |
| D1 | v1.0.0 采用 React + TypeScript + Vite | 从静态 MVP 进入正式前端工程 |
| D2 | 采用 monorepo 目录 | `apps/web` + `packages/*` + `content` |
| D3 | 内容包继续使用 JSON 文件 | 不接后端，不做内容后台 |
| D4 | `content-runtime` 负责内容加载和查询 | 前端页面不直接拼内容文件 |
| D5 | `game-core` 负责判分、解锁、错题、Boss、勋章 | 游戏规则不依赖 DOM |
| D6 | UI 组件只接收 props | 不写死教材、题目或内容包路径 |
| D7 | 保持 localStorage，补迁移策略 | 兼容 v0.2.0 数据 |
| D8 | v1.0.0 不新增玩法 | 优先工程化和正式体验稳定性 |

## 7. 迁移原则

### 7.1 保持用户体验稳定

v1.0.0 不以重设计为目标。当前页面结构、文案和学习流程应尽量保持一致。

允许优化：

- 组件边界更清楚。
- 样式拆分更合理。
- 移动端稳定性更好。
- 加载失败提示更清楚。

不允许趁机做：

- 新地图结构。
- 新关卡机制。
- 新奖励系统。
- 新内容包。

### 7.2 保持内容包兼容

v1.0.0 必须继续读取：

```text
content/math/bsd/grade-2/semester-2/unit-1-division/
```

内容包版本：

```text
math.bsd.g2.s2.unit-1-division@0.1.0
```

如内容 Schema 需要增强，应优先兼容旧字段，不直接破坏 v0.2.0 内容包。

### 7.3 保持学习记录兼容

v1.0.0 必须兼容：

```text
dataVersion: "0.2.0"
contentPackageId: "math.bsd.g2.s2.unit-1-division"
contentVersion: "0.1.0"
```

如果 v1.0.0 提升数据结构，目标数据版本建议为：

```text
dataVersion: "1.0.0"
```

迁移必须保留：

- 昵称。
- 当前进度。
- 已通过关卡。
- 已解锁线索。
- 错题记录。
- 勋章记录。

## 11. 验收标准

v1.0.0 发布前必须满足：

- React/Vite 应用可本地启动。
- 静态构建可通过。
- 当前除法内容包可加载。
- 4 个普通关卡可完成。
- 线索和知识卡可解锁。
- 错题可进入悬案馆并复习。
- Boss 可解锁、失败、重试、结案。
- 勋章可发放。
- 刷新后数据不丢失。
- v0.2.0 localStorage 可迁移或安全读取。
- 内容校验脚本通过。
- 无 P0 / P1 Bug。

## 12. 风险与控制

| 风险 | 控制 |
| --- | --- |
| 工程化 PR 太大 | 拆成多个 Issue 和 PR |
| 迁移时破坏现有流程 | 先迁移 runtime 和 game-core，再迁 UI |
| TypeScript 类型过度设计 | 先覆盖 v0.2.0 已有 Schema |
| 内容路径混乱 | content-runtime 统一读取 |
| localStorage 丢失 | 单独 Issue 做迁移测试 |
| UI 重构失控 | v1.0 不重设计，只组件化 |

## 13. Issue / PR 顺序

```text
Issue 1：搭建 React + TypeScript + Vite 工程骨架
-> Issue 2：建立 content-schema 与内容类型
-> Issue 3：建立 content-runtime

```
