## 2. 文档目的

本文档定义《动物侦探城》的产品版本号、内容包版本号、Git tag、GitHub Release、版本冻结、Issue / Branch / PR 开发流程、发布检查、回滚和复盘规则。

目标是保证：

- 产品版本可追踪。
- 内容包版本可独立演进。
- 每次变更都有 Issue、分支、PR 和验收记录。
- 不直接在 `main` 上开发。
- 发布前范围明确、质量可控、回滚有依据。

## 3. 已确认关键决策

| 编号 | 决策 | 说明 |
| --- | --- | --- |
| D1 | Git tag 与产品版本一致 | 例如 `v0.2.0` |
| D2 | 内容包采用独立版本号 | 例如 `math.bsd.g2.s2.unit-1-division@0.1.0` |
| D3 | 每个产品版本 tag 创建 GitHub Release | Release 记录产品、内容包、已知问题和回滚方案 |
| D4 | 每个版本启动前冻结范围 | 冻结目标、功能、内容、不做清单、验收标准 |
| D5 | Patch 只用于修复和小优化 | 不新增功能、新关卡或结构变化 |
| D6 | v0.2.0 只做内容抽取和结构化 | 不新增玩法、不扩展新案件、不接后端 |
| D7 | 发布前必须通过检查清单 | 文档、测试、内容校验、核心流程、Release Notes、回滚 |
| D8 | 开发过程必须通过 Issue、独立分支和 PR 管理 | 禁止直接在 `main` 开发 |

## 4. 产品版本号规则

### 4.1 格式

产品版本采用：

```text
MAJOR.MINOR.PATCH
```

Git tag 使用：

```text
vMAJOR.MINOR.PATCH
```

示例：

```text
v0.1.0
v0.2.0
v1.0.0
v1.1.0
```

### 4.2 版本含义

| 类型 | 含义 | 示例 |
| --- | --- | --- |
| MAJOR | 产品阶段或架构发生重大变化 | v1.5.0 -> v2.0.0 |
| MINOR | 新增完整功能、内容包或阶段能力 | v1.0.0 -> v1.1.0 |
| PATCH | Bug、文案、样式或小修正 | v1.0.0 -> v1.0.1 |

### 4.3 当前版本规划

| 产品版本 | 名称 | 目标 |
| --- | --- | --- |
| v0.1.0 | 静态 MVP 归档版 | 保存当前静态 MVP 和项目文档 |
| v0.2.0 | 内容抽取版 | 把内容从 `app.js` 抽成 JSON 内容包 |
| v1.0.0 | 除法正式体验版 | 打磨首个单元正式体验 |
| v1.1.0 | 除法内容扩展版 | 扩展同单元案件和题库 |
| v1.5.0 | 方向与位置扩展版 | 验证跨单元能力 |
| v2.0.0 | 多单元地图版 | 支持二年级下册多个数学单元 |
| v2.5.0 | 家长反馈版 | 增加轻量家长查看 |
| v3.0.0 | 平台化内容版 | 内容后台、多年级、多学科 |

## 5. 内容包版本号规则

### 5.1 格式

内容包版本独立于产品版本，格式：

```text
contentPackageId@MAJOR.MINOR.PATCH
```

示例：

```text
math.bsd.g2.s2.unit-1-division@0.1.0
math.bsd.g2.s2.unit-1-division@0.1.1
math.bsd.g2.s2.unit-1-division@0.2.0
```

### 5.2 产品版本引用内容包版本

产品发布时必须记录使用的内容包版本：

```text
product: v0.2.0
contentPackage: math.bsd.g2.s2.unit-1-division@0.1.0
```

### 5.3 内容包版本升级规则

| 类型 | 触发 |
| --- | --- |
| MAJOR | 内容结构或知识图谱重大变化，可能影响历史记录 |
| MINOR | 新增题目、知识卡、关卡、Boss 或复习规则 |
| PATCH | 修正错别字、个别答案、解析或轻微文案 |

## 6. GitHub 开发流程

### 6.1 总流程

所有开发、文档和内容变更都必须遵守：

```text
创建 Issue
-> 明确范围和验收标准
-> 从 main 创建独立分支
-> 修改文档 / 内容 / 代码
-> 本地自测
-> 提交 Pull Request
-> Review
-> 合并 main
-> 必要时创建 tag 和 Release
```

禁止：

```text
直接在 main 上开发
无 Issue 直接开 PR
无验收标准合并 PR
把无关变更混在同一个 PR
```

### 6.2 Issue 规则

每个 Issue 必须说明：

```text
背景：
目标：
范围：
不做：
验收标准：
关联文档：
关联版本：
```

Issue 类型建议：

| 类型 | 用途 |
| --- | --- |
| docs | 文档 |
| product | 产品需求 |
| content | 内容资产 |
| design | UI / UX / 资产 |
| frontend | 前端 |
| backend | 后端 |
| bug | 缺陷 |
| test | 测试 |
| release | 发布 |

### 6.3 分支规则

分支从 `main` 创建。

命名：

```text
docs/<issue-id>-<topic>
feature/<issue-id>-<module>
fix/<issue-id>-<bug>
content/<issue-id>-<case-or-unit>
test/<issue-id>-<scope>
release/<version>
```

示例：

```text
docs/12-release-flow
feature/23-wrong-review
content/31-carrot-badge-package
fix/42-mobile-option-layout
release/v0.2.0
```

### 6.4 PR 规则

每个 PR 必须包含：

- 关联 Issue。
- 变更目的。
- 影响范围。
- 验证方式。
- 是否影响文档。
- 是否影响内容包。
- 是否影响数据结构。
- 是否影响已有学习记录。

PR 合并前必须：

- 自测通过。
- 无 P0 / P1。
- 相关文档同步。
- 内容变更经过对应审核。
- 涉及 UI 时提供截图或说明。

## 7. 版本冻结规则

每个版本启动前必须冻结：

```text
版本目标
功能范围
内容范围
不做清单
验收标准
预计发布时间
负责人
```

冻结后只允许：

- P0 / P1 Bug 修复。
- 不改变范围的文案或样式修正。
- 发布所需的测试和文档补充。

冻结后不允许：

- 新增玩法。
- 新增关卡。
- 新增未评审内容包。
- 调整核心数据结构。
- 插入未评估需求。

## 8. Patch 规则

Patch 可用于：

- 错别字。
- 个别题目答案或解析修正。
- 轻微 UI 问题。
- 小 Bug。
- 不改变主流程的小优化。

Patch 不允许：

- 新增功能。
- 新增关卡。
- 新增内容包结构。
- 改动核心流程。
- 改动数据模型主干。

## 9. v0.2.0 范围锁定

v0.2.0 目标：

```text
内容抽取和结构化。
```

v0.2.0 做：

- 抽出 JSON 内容包。
- 增加内容包 manifest。
- 增加知识图谱文件。
- 增加题目、知识卡、线索、Boss、勋章 JSON。
- 增加内容校验脚本。
- 本地学习记录增加版本字段。

v0.2.0 不做：

- 新玩法。
- 新案件。
- 新单元。
- 后端 API。
- 账号系统。
- 内容后台。
- 家长端。

## 10. Git Tag 规则

### 10.1 创建时机

只有当产品版本准备发布时才创建 tag。

```text
git tag -a v0.2.0 -m "v0.2.0 content extraction"
git push origin v0.2.0
```

### 10.2 Tag 与 Release

每个产品 tag 必须对应一个 GitHub Release。

示例：

```text
tag: v0.2.0
GitHub Release: v0.2.0
```

## 11. GitHub Release 规则

Release 必须记录：

```text
版本号：
发布日期：
版本目标：
产品范围：
内容包版本：
新增：
优化：
修复：
不做范围：
验证方式：
已知问题：
是否影响学习记录：
回滚方案：
下一版本计划：
```

## 12. 发布前检查

发布前必须通过：

```text
文档已更新
功能测试通过
内容校验通过
核心流程通过
无 P0 / P1 Bug
版本号确认
内容包版本确认
Release Notes 准备
回滚方案准备
```

如果是内容包发布，还必须确认：

```text
题目答案和解析已审核
知识点映射已审核
Boss 失败诊断已审核
儿童可读性已审核
```

## 13. 回滚规则

### 13.1 产品回滚

产品发布后出现 P0 / P1 问题：

```text
暂停发布
记录问题
回滚到上一个稳定 tag
创建 hotfix Issue
修复后发布 PATCH
```

### 13.2 内容包回滚

内容包出现问题：

| 问题 | 处理 |
| --- | --- |
| 个别错别字 | Patch 修订 |
| 个别解析不清 | Patch 修订 |
| 个别答案错误 | 立即修正并发布 Patch |
| 知识点映射错误 | 暂停相关内容，重新审核 |
| 大面积教材不一致 | 回滚到上一内容包版本 |

## 14. 发布复盘

每个版本发布后 3-7 天内复盘。

复盘内容：

```text
版本目标是否达成？
范围是否控制住？
是否出现未评估需求插入？
核心流程是否稳定？
儿童测试反馈如何？
内容质量问题有哪些？
Bug 主要来自哪里？
哪些流程可以提前避免？
下一版本应该继续、停止、开始做什么？
```

复盘输出：

- 数据总结。
- 用户反馈。
- Bug 总结。
- 内容问题。
- 改进项。
- 下一版本建议。
