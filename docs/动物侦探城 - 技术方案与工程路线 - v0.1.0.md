## 2. 文档目的

本文档定义《动物侦探城》的技术演进路线、工程分层、前端技术栈、内容包加载、游戏核心逻辑、数据存储、内容校验、部署和后端预留方向。

目标是让项目从当前静态 MVP 平滑演进到：

- 内容可配置。
- 前端可组件化。
- 游戏逻辑可复用。
- 学习数据可迁移。
- 后续可接入后端和内容后台。

## 4. 工程演进路线

### 4.1 阶段总览

| 阶段 | 工程形态 | 目标 |
| --- | --- | --- |
| v0.1.0 | 静态 MVP 归档 | 保留 `index.html / styles.css / app.js`，完成项目资产保存 |

v2.0.0 开始评估：

- 用户系统。

v3.0.0 开始评估：

## 5. 当前 v0.1.0 状态

当前工程是静态单页 MVP：

```text
index.html
styles.css
app.js
localStorage
```

当前优势：

- 能快速验证完整学习闭环。
- 无构建成本。
- 可直接打开或静态服务运行。
- 适合早期用户测试。

当前限制：

- 内容和逻辑混在 `app.js`。
- 教材、知识点、题目、关卡写死。
- 不支持内容包切换。
- 不支持 Schema 校验。
- 不支持内容版本管理。
- 不支持复杂学习诊断。

## 7. v1.0.0 技术目标：前端工程化

### 7.1 技术栈

确认采用：

```text
React
TypeScript
Vite
```

### 7.2 为什么不是一开始用 Next.js

当前项目核心是互动学习应用，不是 SEO 官网。

v1.0 阶段优先：

- 组件化。
- 内容 runtime。
- 游戏状态管理。
- 移动端体验。
- 静态部署。

如果未来需要服务端渲染、复杂账号、后台路由或服务端数据加载，再评估 Next.js 或其他全栈框架。

### 7.3 v1.0 目标

- 前端组件化。
- 内容包 runtime 化。
- game-core 独立。
- UI 组件独立。
- 本地状态有版本和迁移。
- 保持静态部署能力。

## 8. 推荐 monorepo 结构

### 8.1 目录结构

```text
apps/
  web/
packages/
  content-schema/
  content-runtime/
  game-core/
  ui/
content/
docs/
scripts/
```

### 8.2 模块职责

| 模块 | 职责 |
| --- | --- |
| apps/web | 学生端应用，负责路由、页面和整体状态连接 |
| packages/content-schema | 内容对象类型、JSON Schema、校验规则 |
| packages/content-runtime | 加载内容包、解析引用、提供查询 API |
| packages/game-core | 判分、解锁、错题、Boss、勋章、复习规则 |
| packages/ui | Button、QuestionCard、ClueCard、WrongCaseCard、BadgeCard 等组件 |
| content | 教材、知识图谱、题库、案件、知识卡、线索、Boss |
| scripts | 内容校验、内容迁移、构建辅助 |
| docs | 产品、架构、内容、流程和测试文档 |

### 8.3 依赖方向

推荐依赖方向：

```text
apps/web
-> packages/ui
-> packages/game-core
-> packages/content-runtime
-> packages/content-schema
-> content
```

禁止：

- `content` 依赖应用代码。
- `content-schema` 依赖 UI。
- `game-core` 依赖具体 DOM 或浏览器 UI。
- `ui` 内部写死具体教材和题目。

## 9. 内容 runtime

### 9.1 作用

`content-runtime` 负责把 JSON 内容包转换成应用可消费的数据。

它应该提供：

- 加载内容包。
- 校验内容包版本。
- 根据 ID 查找教材节点、知识点、关卡、题目、知识卡、Boss。
- 解析引用关系。
- 提供当前案件的学习路径。

### 9.2 API 草案

```ts
loadContentPackage(packagePath): ContentPackage
getCase(caseId): Case
getLevel(levelId): Level
getQuestionsByLevel(levelId): Question[]
getKnowledgeCard(cardId): KnowledgeCard
getBossTask(bossTaskId): BossTask
getReviewRuleByWrong(questionId): ReviewRule
validateContentPackage(package): ValidationResult
```

## 10. game-core

### 10.1 作用

`game-core` 负责学习游戏规则，不负责 UI。

它应该处理：

- 答案判定。
- 关卡通过。
- 线索解锁。
- Boss 解锁。
- Boss 通过与失败诊断。
- 错题记录。
- 掌握状态更新。
- 勋章发放。
- 复习路径推荐。

### 10.2 API 草案

```ts
checkAnswer(question, userAnswer): AnswerResult
finishLevel(level, answerResults): LevelResult
unlockClues(progress, levelResult): Progress
canStartBoss(progress, caseConfig): boolean
finishBoss(bossTask, stepResults): BossResult
recordWrongQuestion(answerResult): WrongRecord
updateMastery(answerResult): MasteryState
awardBadges(progress, event): BadgeRecord[]
getReviewPlan(wrongRecord): ReviewPlan
```

### 10.3 规则原则

- 规则输入来自内容包。
- 规则输出是数据结果。
- 不直接操作 DOM。
- 不直接读写 localStorage。
- 不写死教材版本。

## 12. 状态管理

### 12.1 v0.2 / v1.0 本地状态

先使用浏览器本地存储：

```text
localStorage
```

必须带：

```text
dataVersion
contentPackageId
contentVersion
```

### 12.2 本地数据结构

```text
UserProfile
Progress
AnswerRecord
WrongRecord
MasteryState
BossAttempt
BadgeRecord
```

### 12.3 迁移策略

当数据结构变化时，需要：

```text
currentDataVersion
targetDataVersion
migration function
fallback reset strategy
```

不能无提示清空孩子学习记录，除非是明确测试环境或用户主动重置。

## 13. 内容校验脚本

### 13.1 命令建议

```text
validate-content
```

v0.2.0 可以先用 Node.js 脚本实现。

### 13.2 校验内容

必须检查：

- manifest 是否完整。
- ID 是否唯一。
- 题目是否绑定教材、知识点、能力点、易错点。
- 关卡引用的题目是否存在。
- 知识卡引用的知识点是否存在。
- 线索引用的关卡是否存在。
- Boss 步骤题是否存在。
- Boss 目标知识点和能力点是否存在。
- 复习规则引用是否完整。
- 勋章触发规则是否有效。
- 内容状态是否允许发布。

### 13.3 校验结果

```text
pass
warning
error
```

存在 `error` 时禁止发布。

## 16. 安全与隐私原则

### 16.1 v0.x / v1.x

- 不采集真实姓名。
- 不要求手机号。
- 不上传学习数据到服务器。
- 本地数据可重置。
- 不接入支付。

### 16.2 v2.x 以后

如接入账号和后端，需要补充：

- 隐私政策。
- 儿童数据保护方案。
- 家长授权机制。
- 数据删除机制。
- 权限控制。
- 日志脱敏。
