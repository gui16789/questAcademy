## 2. 目标

把当前单内容包体验扩展为可持续接入北师大版二年级数学上、下册全部单元的内容生产体系。

本阶段目标不是一次性生成所有题目，而是先建立：

- 全册单元接入清单。
- 每个单元的内容包 ID 和目录规则。
- 教材节点、知识点、能力点、易错点、题型模板的拆解流程。
- 第二个内容包试点的落地路径。
- 后续批量接入、审核、回归和发布门槛。

## 3. 核心原则

| 编号 | 原则 | 说明 |
| --- | --- | --- |
| P1 | 单元作为内容包粒度 | 每个单元独立审核、校验、回归和发布 |
| P2 | 知识地图先于题目 | 先确认教材节点与知识点，再生产题目和剧情 |
| P3 | 草稿不进默认体验 | 未审核内容包不设为默认，不影响当前除法单元 |
| P4 | 教研复核优先 | 本文单元目录是接入草案，必须按实际教材版本复核 |
| P5 | 先试点后规模化 | 先完成第二个内容包，再复制流程到全册 |

## 7. 第二个内容包试点建议

建议优先试点：

```text
math.bsd.g2.s2.unit-2-direction
```

理由：

- 与当前除法单元差异明显，能验证内容包机制是否真正通用。
- 题目以判断、选择、填空为主，暂时不必引入复杂拖拽。
- 案件包装自然，例如“迷路的快递路线”“地图坐标失踪案”。
- 适合验证地图、方向、位置等非计算类知识点的表达方式。

试点验收：

- 至少 3-4 个知识点。
- 每个知识点 1 个关卡。
- 每关至少 4 道正式题。
- 每个知识点 1 张知识卡。
- 每个核心能力点至少 1 个易错点。
- 1 个 Boss 综合任务。
- `validate:content -- --package math.bsd.g2.s2.unit-2-direction` 通过。
- 前端可通过 `?contentPackageId=math.bsd.g2.s2.unit-2-direction` 加载。
- 浏览器回归通过。

## 8. 每个单元的生产步骤

```text
1. 教研复核单元目录和页码
2. 生成内容包草稿骨架
3. 填写 textbook.json
4. 拆解 knowledge-map.json
5. 设计案件、关卡和 Boss
6. 生产题目、知识卡、线索和复习规则
7. 内容审核
8. 教研审核
9. 儿童可读性审核
10. 登记 content/registry.json
11. 接入 bundled loader 或静态 JSON loader
12. 跑内容校验、类型检查、测试和回归
13. 发布内容包版本
```

生成草稿骨架命令：

```bash
npm run create:content-package -- --id math.bsd.g2.s2.unit-2-direction --title "北师大版二年级下册第二单元：方向与位置"
```

## 9. 单元内容包最小交付标准

| 类型 | 最小数量 | 说明 |
| --- | --- | --- |
| 教材节点 | 3 个 | 按实际教材课时或主题拆分 |
| 知识点 | 3 个 | 每个知识点要有定义、目标和前置知识 |
| 能力点 | 每知识点至少 1 个 | 必须可观察、可出题、可诊断 |
| 易错点 | 每核心能力点至少 1 个 | 必须能关联错题和复习规则 |
| 普通关卡 | 3-5 个 | 通常一个核心知识点一个关卡 |
| 普通题 | 每关至少 4 道 | 建议后续扩展到 6-8 道 |
| 备用题 | 每单元至少 4 道 | 用于复习和内容替换 |
| 知识卡 | 每知识点 1 张 | 短句、例题、方法、常见错误 |
| 线索 | 每关 1 条 | 推动案件，不替代知识解释 |
| Boss | 每单元 1 个 | 综合验证核心知识点 |
| 复习规则 | 每易错点 1 条 | 指向知识卡和题型模板 |

## 11. 风险

| 风险 | 表现 | 处理 |
| --- | --- | --- |
| 教材目录不准确 | 单元名、课时、页码与实际书不一致 | 内容包进入生产前必须教研复核 |
| 草稿内容误发布 | 占位题进入学生体验 | draft 包不设默认，发布前校验审核状态 |
| 题型能力不足 | 图形、时间、统计单元难以表达 | 先用选择/判断/填空试点，必要时扩展题型 |
| 内容包太多后 loader 难维护 | 手工 import JSON 成本高 | 改为静态资源 fetch 或自动生成 import map |
| 审核跟不上生产 | AI 草稿数量大但质量不稳定 | 控制单元节奏，每包必须走三段审核 |
