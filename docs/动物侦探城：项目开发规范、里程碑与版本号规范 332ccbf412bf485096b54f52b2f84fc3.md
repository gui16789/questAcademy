# 动物侦探城：项目开发规范、里程碑与版本号规范

## 1. 文档目的

本文档用于规范《动物侦探城》从 MVP 验证后进入完整项目开发阶段的工作方式，重点解决以下问题：

- 如何避免需求越来越大、项目中途跑偏。
- 如何把“探案式学习”长期稳定地扩展为可持续产品。
- 如何建立内容工厂化流程，持续生产案件、关卡、题目、线索卡和 Boss 挑战。
- 如何制定清晰的里程碑、版本号和发布节奏。
- 如何让产品、教研、设计、开发、测试围绕同一套标准协作。

本文档是项目进入长期开发阶段后的**项目管理总规范**。

---

## 2. 项目总原则

### 2.1 一句话原则

```
先保证学习闭环有效，再扩展游戏世界；先保证小版本可交付，再追求大版本完整。
```

### 2.2 项目核心闭环

所有版本都必须围绕以下主线展开：

```
接案
→ 学习知识点
→ 普通关卡练习
→ 收集线索
→ 错题进入悬案馆
→ 知识点进入线索库
→ Boss 综合挑战
→ 结案
→ 获得勋章
→ 复习与再挑战
```

任何新功能如果不能增强这条闭环，都不能优先进入当前版本。

### 2.3 研发优先级原则

| 优先级 | 原则 |
| --- | --- |
| 第一优先级 | 学习效果 |
| 第二优先级 | 儿童可理解性 |
| 第三优先级 | 完整闭环 |
| 第四优先级 | 游戏趣味 |
| 第五优先级 | 美术表现 |
| 第六优先级 | 扩展玩法 |

### 2.4 不做原则

为了避免项目膨胀，以下功能不得在核心学习闭环稳定前优先开发：

- 排行榜
- 商城
- 复杂宠物养成
- 多人对战
- 大量动画剧情
- 复杂社交
- 未经人工审核的 AI 自动出题
- 与学习结果无关的装饰型系统

## 3. 完整项目六大文档体系

MVP 之后，完整项目建议维护以下 6 类核心文档。

### 3.1 产品愿景与边界文档

用途：

```
定义产品为什么存在、服务谁、解决什么问题、不解决什么问题。
```

必须包含：

- 产品一句话定位
- 目标用户
- 核心价值
- 长期愿景
- 当前阶段边界
- 明确不做清单
- IP 与合规原则
- 成功指标

维护负责人：

```
产品负责人
```

更新频率：

```
大版本启动前更新，日常不频繁修改。
```

---

### 3.2 PRD 产品需求文档

用途：

```
定义当前版本具体要做什么、怎么做、如何验收。
```

必须包含：

- 版本目标
- 用户故事
- 功能清单
- 页面清单
- 流程图
- 数据需求
- 交互规则
- 验收标准
- 不做范围
- 风险说明

维护负责人：

```
产品负责人
```

更新频率：

```
每个版本启动前冻结一版；开发中变更必须走变更流程。
```

---

### 3.3 GDD 游戏设计文档

用途：

```
定义游戏世界观、角色、地图、关卡、奖励、失败反馈和成长体系。
```

必须包含：

- 世界观
- 地图结构
- 角色设定
- 案件流程
- 线索规则
- Boss 规则
- 勋章规则
- 失败反馈机制
- 成长系统
- 游戏化语言规范

维护负责人：

```
产品负责人 + 游戏策划 / 内容策划
```

更新频率：

```
每个新增地图、玩法或成长系统前更新。
```

---

用途：

```
定义教材知识点如何转化为案件、关卡、题目、线索卡、Boss 和错题复习。

- 线索卡模板

```

更新频率：

### 3.6 项目计划、里程碑与版本规范文档

```
定义项目分几个阶段做、每个阶段交付什么、版本号如何命名、上线标准是什么。

## 4. 角色与职责规范

### 4.1 核心角色

| 角色 | 主要职责 |
| --- | --- |
| 产品负责人 | 定义需求、控制范围、维护 PRD、确认验收 |
| 项目负责人 | 控制进度、协调资源、管理风险、组织复盘 |
| 教研负责人 | 拆解知识点、审核题目、保证教育准确性 |
| 内容策划 | 编写案件剧情、线索卡、反馈文案、Boss 剧情 |
| UI/UX 设计 | 设计页面、交互、儿童友好体验 |
| 美术设计 | 设计角色、地图、勋章、插画资产 |
| 前端开发 | 实现页面、交互、答题流程 |
| 后端开发 | 实现数据、题库、进度、错题、勋章系统 |
| 测试 | 功能测试、流程测试、兼容性测试、儿童体验问题记录 |
| 数据分析 | 跟踪完成率、复玩率、错题复习率、Boss 通过率 |

### 4.2 决策原则

如果出现争议，按以下顺序决策：

```
是否影响学习效果
→ 是否影响儿童理解
→ 是否影响主闭环完成
→ 是否影响当前版本上线
→ 是否影响后续扩展
```

## 5. 需求管理规范

### 5.1 需求来源

需求可以来自：

- 用户测试反馈
- 家长反馈
- 教研建议
- 产品规划
- 数据分析
- 开发技术优化
- 运营需求

### 5.2 需求池分类

所有新想法必须先进入需求池，不得直接插入当前开发。

| 状态 | 含义 |
| --- | --- |
| 待评估 | 刚提出，尚未判断价值 |
| 已接受 | 确认有价值，但未排期 |
| 当前版本 | 已进入当前版本范围 |
| 后续版本 | 暂不做，放入未来版本 |
| 已拒绝 | 不符合产品方向或成本过高 |
| 已完成 | 已开发上线 |

### 5.3 需求优先级

| 优先级 | 定义 | 示例 |
| --- | --- | --- |
| P0 | 没有它，核心闭环无法完成 | 答题、判分、错题记录 |
| P1 | 强烈影响体验和学习效果 | 知识卡、Boss 解析 |
| P2 | 有价值，但不影响版本上线 | 更多角色头像 |
| P3 | 锦上添花 | 特效、装饰、彩蛋 |

### 5.4 需求进入当前版本的判断标准

一个需求进入当前版本前，必须回答：

```
1. 它是否服务当前版本目标？
2. 它是否增强学习闭环？
3. 不做它，当前版本是否还能上线？
4. 它是否有明确验收标准？
5. 它是否会明显增加延期风险？
```

如果第 1、2、4 项不能明确回答，原则上不进入当前版本。

## 6. 防跑偏机制

### 6.1 版本范围冻结

每个版本启动时必须冻结以下内容：

- 版本目标
- 功能清单
- 页面清单
- 不做清单
- 验收标准
- 预计上线时间

冻结后，除非出现 P0 级问题，不得随意新增功能。

### 6.2 变更申请机制

开发中如需新增或修改需求，必须填写变更说明。

#### 变更说明模板

```
变更名称：
提出人：
提出时间：
变更原因：
影响模块：
是否影响核心学习闭环：
是否影响上线时间：
是否需要新增设计：
是否需要新增开发：
是否需要新增测试：
不做该变更的影响：
建议处理方式：当前版本 / 下个版本 / 暂不做
最终决策：
```

### 6.3 每周防跑偏检查

每周固定检查以下问题：

```
1. 本周工作是否推进了核心学习闭环？
2. 是否有新增需求进入当前版本？
3. 新增需求是否经过评估？
4. 当前版本是否仍可按计划交付？
5. 是否有功能看起来很酷但不服务学习目标？
6. 是否有内容生产滞后风险？
7. 是否有测试反馈显示孩子理解困难？
```

### 6.4 三条红线

以下情况必须立即暂停扩展，回到版本目标：

```
1. 连续两周没有可演示进展。
2. 当前版本新增需求超过原计划 20%。
3. 核心学习闭环尚未跑通，却开始开发装饰型玩法。
```

### 6.5 后续版本池

所有好想法都可以保留，但必须放入后续版本池，例如：

- 宠物系统
- 排行榜
- 商城
- 语音朗读
- 家长端
- 老师端
- 多人协作破案
- AI 出题
- 动画剧情

原则：

```
先记录，不马上做。
```

## 8. 开发流程规范

### 8.1 版本开发流程

每个版本按以下流程推进：

```
版本立项
→ 需求评审
→ 内容评审
→ 原型设计
→ 技术评审
→ 开发排期
→ 开发实现
→ 联调
→ 测试
→ 修复
→ 验收
→ 灰度测试
→ 正式发布
→ 数据观察
→ 版本复盘
```

### 8.2 开发前必须完成

开发开始前必须具备：

- PRD 已确认
- 不做清单已确认
- 页面原型已确认
- 内容清单已确认
- 题目样例已确认
- 数据结构已确认
- 验收标准已确认
- 版本截止时间已确认

### 8.3 开发任务拆分规则

每个任务必须足够小，建议 1～3 天可完成。

任务必须包含：

```
任务名称
所属模块
需求说明
输入内容
输出结果
验收标准
负责人
预计完成时间
依赖项
```

### 8.4 Definition of Done 完成定义

一个功能只有同时满足以下条件，才算完成：

```
1. 功能按 PRD 实现。
2. 页面和交互符合设计稿。
3. 数据能正确保存和读取。
4. 异常状态有处理。
5. 基础测试通过。
6. 验收标准全部满足。
7. 没有 P0 / P1 级 Bug。
8. 相关文档已更新。
```

## 9. 测试与验收规范

### 9.1 测试类型

| 测试类型 | 说明 |
| --- | --- |
| 功能测试 | 功能是否按需求实现 |
| 流程测试 | 用户能否完成完整闭环 |
| 内容测试 | 题目、解析、知识点是否正确 |
| 儿童体验测试 | 孩子是否看得懂、愿意继续 |
| 兼容性测试 | 不同设备和浏览器是否可用 |
| 数据测试 | 进度、错题、勋章是否正确保存 |
| 回归测试 | 修复 Bug 后是否影响已有功能 |

### 9.2 验收角色

| 模块 | 验收人 |
| --- | --- |
| 产品流程 | 产品负责人 |
| 教学内容 | 教研负责人 |
| 视觉体验 | UI/UX 负责人 |
| 技术实现 | 技术负责人 |
| 整体上线 | 项目负责人 |

### 9.3 Bug 等级

| 等级 | 定义 | 示例 |
| --- | --- | --- |
| P0 | 阻断核心流程 | 无法答题、无法进入案件 |
| P1 | 严重影响体验或数据 | 错题未保存、Boss 无法结案 |
| P2 | 一般问题 | 文案错字、样式轻微错位 |
| P3 | 轻微优化 | 动效不够流畅、按钮可更美观 |

### 9.4 上线前检查清单

```
核心流程可完整跑通
题目答案和解析已审核
错题可进入悬案馆
错题可重新作答
线索可正常发放
Boss 可正常解锁
结案状态可保存
勋章可正常发放
刷新页面数据不丢失
移动端和桌面端可用
无 P0 / P1 Bug
版本号已确认
发布说明已准备
```

## 10. 项目计划与里程碑

### 10.1 阶段总览

| 阶段 | 阶段名称 | 目标 |
| --- | --- | --- |
| Phase 0 | MVP 验证 | 验证探案式学习闭环是否成立 |
| Phase 1 | V1.0 正式版 | 打磨首个单元完整体验 |
| Phase 2 | V1.1 内容扩展 | 扩展除法单元内容和复习能力 |
| Phase 3 | V1.5 单元扩展 | 扩展到第二单元“方向与位置” |
| Phase 4 | V2.0 多单元体系 | 建立二年级下册多单元地图 |
| Phase 5 | V2.5 家长反馈 | 增加轻量家长查看与学习报告 |
| Phase 6 | V3.0 平台化 | 扩展多学科、多年级和内容后台 |

---

### 10.2 Phase 0：MVP 验证

目标：

```
完成一个可独立体验的探案式数学学习闭环。
```

交付物：

- 1 个案件
- 4 个普通关卡
- 1 个 Boss
- 悬案馆
- 线索库
- 勋章馆
- 约 28 道题

成功标准：

```
70% 以上孩子完成首个案件。
60% 以上孩子愿意继续玩下一案。
70% 以上家长认为形式有学习价值。
Boss 通过率在 50%～80% 之间。
```

阶段门：

```
如果 MVP 成功指标未达到，不进入大规模扩展，先优化核心闭环。
```

## 11. 版本号规范

### 11.1 版本号格式

采用语义化版本号：

```
主版本号.次版本号.修订号
```

格式：

```
MAJOR.MINOR.PATCH
```

示例：

```
1.0.0
1.1.0
1.1.1
2.0.0
```

---

### 11.2 版本号含义

| 类型 | 含义 | 示例 |
| --- | --- | --- |
| 主版本号 MAJOR | 产品阶段或架构发生重大变化 | 1.0.0 → 2.0.0 |
| 次版本号 MINOR | 新增较完整功能或内容，但不破坏原功能 | 1.0.0 → 1.1.0 |
| 修订号 PATCH | Bug 修复、文案修正、小优化 | 1.1.0 → 1.1.1 |

---

### 11.3 版本命名规则

建议结合版本号和阶段名称：

```
v0.1.0 MVP 原型版
v0.2.0 MVP 测试版
v1.0.0 除法正式版
v1.1.0 除法内容扩展版
v1.5.0 方向与位置扩展版
v2.0.0 多单元地图版
v2.5.0 家长反馈版
v3.0.0 平台化内容版
```

---

### 11.4 版本阶段定义

| 版本 | 阶段 | 定义 |
| --- | --- | --- |
| v0.x.x | MVP 阶段 | 用于验证核心闭环，不追求完整 |
| v1.x.x | 单学科单年级阶段 | 聚焦二年级下册数学 |
| v2.x.x | 多单元产品阶段 | 扩展多个单元和学习地图 |
| v3.x.x | 平台化阶段 | 支持内容后台、多年级、多学科 |

---

### 11.5 版本发布规则

每个版本发布前必须具备：

```
1. 版本号已确认。
2. 发布范围已确认。
3. 不做清单已确认。
4. 验收标准全部通过。
5. 无 P0 / P1 Bug。
6. 教研内容已审核。
7. 发布说明已完成。
8. 回滚方案已准备。
```

---

### 11.6 Patch 版本规则

以下情况只升级 Patch：

- 修复错别字
- 修复样式问题
- 修复个别题目答案或解析
- 修复不影响核心流程的小 Bug
- 优化少量反馈文案

示例：

```
v1.0.0 → v1.0.1
```

---

### 11.7 Minor 版本规则

以下情况升级 Minor：

- 新增案件
- 新增题目批次
- 新增勋章
- 新增知识卡
- 新增复习流程
- 新增小型功能模块

示例：

```
v1.0.0 → v1.1.0
```

---

### 11.8 Major 版本规则

以下情况升级 Major：

- 新增多个单元体系
- 新增家长端或老师端
- 新增内容管理后台
- 架构发生重大变化
- 从单学科扩展到多学科
- 从单年级扩展到多年级

示例：

```
v1.5.0 → v2.0.0
```

---

### 11.9 预发布版本规则

测试版本可使用以下后缀：

```
-alpha
-beta
-rc
```

示例：

```
v1.0.0-alpha.1
v1.0.0-beta.1
v1.0.0-rc.1

含义：

| 后缀 | 含义 |
| --- | --- |

| beta | 小范围用户测试 |
| rc | 候选发布版本 |

## 12. 里程碑验收机制

### 12.1 阶段门机制

每个阶段结束后必须进行阶段门评审。

评审问题：

```
1. 本阶段目标是否达成？
2. 核心学习闭环是否更稳定？
3. 内容生产是否可复制？
4. 儿童测试反馈是否正向？
5. 家长是否认可？
6. 技术债是否可控？
7. 是否可以进入下一阶段？
```

### 12.2 进入下一阶段的条件

必须满足：

```
当前阶段核心目标已达成。
无阻断性 Bug。
关键数据达到最低成功指标。
内容审核通过。
下一阶段范围明确。
资源和时间可支持。
```

### 12.3 不进入下一阶段的情况

以下情况应暂停扩展：

```
孩子无法理解主流程。
错题复习无人使用。
Boss 难度明显不合理。
家长不认可学习价值。
内容生产严重依赖个人灵感，无法模板化。
技术结构无法支撑扩展。
```

## 17. 总结

《动物侦探城》完整项目开发阶段的关键，不是一次性做出大而全的系统，而是建立一套可以长期复用的开发和内容生产机制。

项目必须始终坚持：

```
小步快跑
版本冻结
内容模板化
学习目标优先
数据验证决策
新想法进需求池
每个阶段都可验收
```

只要这套机制稳定运行，项目就能从一个 MVP，逐步扩展为完整的儿童游戏化学习产品，而不容易中途跑偏或烂尾。
